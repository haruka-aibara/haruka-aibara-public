# コードレビューはまだ必要か

「AI でコードを書く量が増えた。でも PR のレビューを必ず人間が通すと、スケールするにつれて人間がボトルネックになる」——この問いに正直に向き合う。

結論から言う。**従来型のコードレビューはスケールしない。人間の判断を投入するポイントを変える必要がある。**

---

## なぜ従来型のレビューがスケールしないか

AI を使う開発者は PR を **98% 多く作成する**。同時に **PR のレビュー時間は 91% 増加する**。

コードを書く速さが上がるほど、レビューの詰まりも比例して増える。これは構造的な問題だ。人間のレビューキャパシティは増えない。

さらに都合が悪いことに、AI が生成したコードのレビューは人間が書いたコードより**認知的にきつい**。OCaml のオープンソースメンテナーが大規模な AI 生成 PR を却下した理由はこれだ——「レビューに必要なコストをプロジェクトが負担できない」。

「じゃあシニアが頑張ればいい」という解決策はスケールしない。スループットが人の数で上限が決まる。

---

## 「レビューがないと品質が保てない」は本当か

コードレビューが標準的なプラクティスになったのは 2012〜2014 年頃。それほど歴史が深いわけではない。当時の課題解決手段として発明されたもので、前提条件が変われば手段も変わる。

品質を守る手段としてのレビューには、実は構造的な弱点がある：

- レビューは**事後的**——コードが書かれた後にチェックする
- レビュアーは**コンテキスト不足**——なぜその実装にしたかを把握しきれない
- 大量の PR では**rubber-stamp（形式的な承認）**が起きる

「レビューがあるから品質が保たれている」は、本当かどうか再評価する価値がある。

---

## 人間の判断を「上流」に移す

latent.space の "How to Kill the Code Review" が提唱するアプローチ：

**コードではなく、仕様・計画・受け入れ条件をレビューする。**

```
従来：コードが書かれた → 人間がコードをレビュー → マージ

新しいアプローチ：
  人間が仕様・受け入れ条件を定義する（上流）
       ↓
  AI が実装する
       ↓
  自動化された検証が通過したら → マージ
```

「このコードは正しく書かれているか」ではなく「この仕様は正しく定義されているか」が人間の判断ポイントになる。

これは[チケット管理](AIエージェント時代のチケット管理.md)の話とつながる。AI エージェントが実行できるチケットを書くということは、同時に「何を達成したら完了か」を人間が明確化するということだ。

---

## 自動化された検証レイヤーで品質を担保する

人間のレビューが薄くなる分を、決定論的な検証で補う。「人間の勘」ではなく、**通過条件を明示したゲート**に置き換える。

### テスト（行動定義）

BDD（振る舞い駆動開発）でテストが「仕様そのもの」になる。

```gherkin
Feature: ユーザー登録

  Scenario: 正常登録
    Given メールアドレスが未登録の状態
    When POST /api/users に正しいデータを送る
    Then ステータス 201 を返す
    And ユーザーが作成される

  Scenario: 重複メールアドレス
    Given メールアドレスがすでに登録済みの状態
    When POST /api/users に同じメールを送る
    Then ステータス 409 を返す
```

このテストが通れば、コードが「仕様通りに動く」ことは機械的に証明される。

### 型チェック・Lint（構文レベル）

人間がコードを見て「型が合ってない」「null が渡る可能性がある」を指摘するより、型チェックを CI で必須にした方が確実で速い。

```yaml
# PR マージ前に必ず通過が必要なチェック
- npm run typecheck     # TypeScript 型エラー
- npm run lint          # ESLint
- npm test -- --coverage  # テストカバレッジ閾値
```

### セキュリティ・依存関係（自動スキャン）

AI が存在しないパッケージ名を推奨することがある（Slopsquatting）。AI 生成コードの約 20% が存在しないパッケージを参照するという研究がある。これは人間が見て気づくより、ツールで検出する方が確実。

```yaml
- npm audit --audit-level=high    # 既知の脆弱性
- npm install --dry-run           # パッケージが実在するか
```

### 複数エージェント検証

コードを書くエージェントとは別のエージェントに、動作確認・セキュリティチェック・テスト作成を担当させる。同一の AI が「書いて自分でチェック」するのではなく、**役割を分離することで相互検証になる**。

---

## 「Ship fast, observe everything, revert faster」

コードレビューでバグを全部防ごうとするのをやめ、本番での問題を素早く検知・切り戻せる仕組みを作る。

フィーチャーフラグと組み合わせると：

1. コードは自動検証を通れば main にマージ
2. 本番にデプロイするがフィーチャーフラグで OFF（ユーザーには届かない）
3. カナリアリリースで 1% → 10% → 100% と段階的に公開
4. CloudWatch アラームが異常を検知したら自動ロールバック

このモデルでは「レビューで事前に防ぐ」から「観測して素早く戻す」にシフトする。詳細は[スクラムチームのデプロイ](スクラムチームのデプロイ.md)を参照。

---

## 何かあるとしたら残すレビュー

全ての人間レビューをゼロにするという話ではない。自動化が代替できないものに絞る。

**残す価値があるレビュー：**
- アーキテクチャの大きな変更（DB スキーマ変更、サービス間の依存追加など）
- 仕様の曖昧さが残っているチケットの実装（上流の定義が不十分だったケース）
- セキュリティ上の重要な変更（認証・認可ロジック）

**自動化に任せていいもの：**
- スタイル・フォーマット（Lint / Prettier）
- 型安全性（TypeScript / mypy）
- 既知のセキュリティパターン（SAST ツール）
- 依存関係の妥当性（npm audit / pip-audit）
- 機能の動作（テスト）

---

## 移行の進め方

一気にレビューをなくすのではなく、段階的に自動化で置き換えながら縮小する。

```
Phase 1: 自動化を整備する
  - CI でテスト・型チェック・セキュリティスキャンを必須にする
  - テストカバレッジの閾値を設定する

Phase 2: レビューの対象を絞る
  - スタイル・Lint の指摘はレビューコメントに含めない（自動化済みのため）
  - アーキテクチャ変更のみ人間レビューを必須とするルールに変える

Phase 3: 本番観測を整備する
  - フィーチャーフラグとカナリアデプロイを導入する
  - アラームとロールバック自動化を設定する

Phase 4: レビューを任意にする
  - 自動検証が通れば任意でマージ可能にする
  - 複雑な変更は書いた側が自分でレビューを依頼する
```
