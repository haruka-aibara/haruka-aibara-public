# コードレビューゼロへのロードマップ（Terraform + Python Lambda 編）

<!-- TODO: 調査エージェントの結果で補強予定 -->

「GitHub で Terraform と Python Lambda を管理しているチームで、コードレビューをゼロにしたい」——[理由はこちら](コードレビューはまだ必要か.md)。では具体的にどうすれば安全に実現できるか。

前提として明確にしておく。**コードレビューをゼロにできるのは「人間が prod に直接触れない構造」が整っているとき**だ。AWS アカウント構成がその基盤になる。

---

## コードレビューは何を守っているのか、本当に考えたことがあるか

「レビューをなくす」と言うと不安になる。その不安を正直に分解してみる。

**「バグを見つけるため」** → そのバグ、テストが先に気づかなかったのはなぜか。テストが通っているのに「このケースは考えた？」と言われるなら、それはテストが足りていないということ。テストを直すべきで、レビューで補うのは場当たり的だ。

**「セキュリティ問題を防ぐため」** → bandit は全ファイルを3秒で検査する。tfsec は Terraform の全モジュールを見る。疲れないし、見落とさない。人間のレビュアーが `import subprocess` の使い方を行 47 で見逃すことは普通にある。機械は見逃さない。

**「コードの一貫性を保つため」** → ruff は 0.1 秒で直す。型チェックも mypy が全パスを追う。人間がやる必要はない。

**「設計の問題を見つけるため」** → これが唯一の本命だ。ただしここに落とし穴がある。

コーディング中に設計が変わるのは普通のことだ。ゴールだけ見えている状態で実装を始めて、書きながら「あ、こっちのほうがいいな」と気づく。それは設計力があることの証拠であって、問題でもなんでもない。

問題は「そのとき何をするか」だ。

**PR レビューという形式でやると何が起きるか：**
- コードを全部書き終わってから PR を出す
- レビュアーが「こっちの設計のほうが良かったと思います」とコメントする
- もうコードは書き終わっている。変更コストが高い
- 書き直す気力があれば直すが、「まあいいか」になることも多い

**コーディング中にリアルタイムで相談したら何が起きるか：**
- 設計の迷いが出た瞬間に Slack で「AとBどっちが良い？」とスクショを投げる
- 5分で返ってくる
- まだコードを書いている最中なので変更コストがゼロに近い

つまり PR レビューは、**設計の相談をする場として最悪のタイミング**だ。コードを書き終わった後に非同期で指摘されても、変更コストが高すぎて機能しない。設計の相談はコーディング中にリアルタイムでやるほうが、速くて安い。

**コードレビューは品質を守る仕組みではなく、「リアルタイムに相談する文化がない」ときの、遅くて高コストな補完手段だった。**

### 機械 vs 人間：何が得意か

| チェックの種類 | 機械 | 人間のレビュアー |
|---|---|---|
| テストで検出できるバグ | CI が必ずブロック | 見落とすことがある |
| セキュリティパターン（SAST） | 全ファイルを毎回検査 | 集中力・時間に依存 |
| フォーマット・スタイル | 0.1秒で完璧に | 指摘するのが気まずい場合もある |
| 依存ライブラリの脆弱性 | pip-audit が自動チェック | ほぼ気づかない |
| 設計の善し悪し | 苦手 | 得意（ただし PR より Slack の方が速い） |

機械が得意なことは機械に任せ、人間が得意なことは人間がリアルタイムでやる。その構造が整えば、PR レビューというステップに人間が介在する理由はなくなる。

## 「早い段階で意思疎通する」ために変える3つの習慣

PR レビューをなくすとき、設計の質が落ちないか不安になるのは自然だ。その不安に答えるのは「自動化」ではなく**フィードバックループを早める習慣**だ。

### 1. 小さく作って早く見せる

大きい PR は問題が多い。

- 500行の PR → レビュアーは1時間かけて読もうとするが、実際には流し読みになる
- 設計上の賭けが大きい（間違っていたときの手戻りが大きい）
- 「全部見た」という錯覚が生まれる

小さい PR は全部逆になる。1日以内にマージできる単位で作ると、「設計の問題が起きる確率」が構造的に低くなる。1日分の判断ミスは修正が軽い。

**重要なのは「承認を得るために見せる」のではなく、「方向性を共有するために早く見せる」こと。** Draft PR を早い段階で開いて「こういう方向で進めます」と伝えるだけで、ズレは早期に解消できる。

### 2. 作業中に声を出す（Working out loud）

設計の迷いが出た瞬間に Slack に投げる。

> 「S3 への書き込みを実装中。presigned URL と直接 PutObject で迷ってる。Lambda 経由でファイルサイズが大きいとタイムアウトリスクあるから presigned URL にしようと思うが、何か見落としある？」

これに5分で返ってくるなら、PR コメントより10倍速く、変更コストが低い状態で判断できる。「コードを全部書き終えた後に非同期でコメントが来る」PR レビューとは根本的に違う。

声を出すことで副次効果もある。返信がなくても「考えを整理した」だけで解決することが多い（ラバーダック効果）。

### 3. 始める前に「何ができたら完成か」を合わせる

コーディング中の設計判断の多くは、**そのタスクのゴールが曖昧なときに迷いが生まれる**。

「このユーザーストーリーが動くこと」ではなく、「このテストが通ること」まで具体化しておくと、実装中の判断軸が生まれる。

```
# タスク開始前に書く acceptance criteria の例
- GET /users/:id が 200 を返す（ユーザーが存在する場合）
- GET /users/:id が 404 を返す（存在しない場合）
- userId が未指定なら 400
- レスポンスタイム p99 < 200ms（DynamoDB local でのテストは除く）
```

この基準があれば、実装中に「A と B、どっちの設計にすべきか」は「どちらがこの基準を満たしやすいか」で判断できる。レビュアーに聞く前に自分で答えが出ることが増える。

### まとめると

| PR レビュー（今まで） | 早期フィードバック（目指す形） |
|---|---|
| 作り終わってから評価する | 作っている途中に一緒に考える |
| 非同期・遅い | リアルタイム・速い |
| 変更コストが高い状態で指摘が来る | 変更コストが低い状態で解決できる |
| 大きい単位でレビュー | 小さい単位で常に共有 |

PR レビューをなくすことは「チェックをなくす」ことではない。**「遅くて高コストなチェックを、速くて安いやり方に置き換える」**ことだ。

## AI コーディングツールが「人間レビュー不要」を加速する

### コード生成速度が変わると、レビューが詰まる

Cursor・Copilot・Claude を使うと、コードを書く速度が体感で 5〜10 倍になる。これはレビューの経済性を根本から壊す。

- 開発者1人が1日に出す PR が 1 本 → 5 本になる
- レビュアーが見るべきコード量が 5〜10 倍になる
- レビュアーを 5〜10 倍雇うことはできない

AI 導入でコードの生産量が増えるほど、**人間レビューがボトルネックになる速度も比例して上がる**。「いつかレビューが詰まる」ではなく、すでに詰まり始めている。

### AI はコードを書きながらすでにレビューしている

Cursor や Copilot は補完を提案するとき、既存コードのパターンを読んで「このコンテキストで正しい書き方」を出している。つまり **AI ペアプロは書いている最中にリアルタイムでレビューが走っている**状態だ。

- 型がずれた書き方を提案しない（型の整合性）
- 既存のユーティリティ関数を再実装しない（重複排除）
- プロジェクトのコーディング規約に沿う（スタイル）

人間が PR を出した後に非同期でコメントする内容の多くは、AI が書いている最中にすでに解決している。

### プッシュ前に AI にチェックさせる

CI が回る前の最後の砦として、AI にセルフレビューをさせることもできる。

```bash
# git commit 前に Claude にチェックさせる例（git hook）
# .git/hooks/pre-push

diff=$(git diff origin/main...HEAD)

claude --print "以下の差分をレビューして。
チェック観点：
- 明らかなバグやロジックの抜け
- セキュリティ上の問題（ハードコードされた認証情報、インジェクション等）
- 既存のパターンから大きく外れていないか

問題なければ 'OK' とだけ返して。問題があれば箇条書きで指摘して。

---
$diff"
```

CI より前の段階で問題を潰せるため、「CI でブロックされて修正して再プッシュ」のループが減る。

### 人間の仕事が「実装のチェック」から「何を作るか」にシフトする

AI がコードを書き、CI が品質を担保し、自動ロールバックが安全網になる。この構造が整うと、**人間にしかできない仕事の重心が変わる**。

| 今まで | AI 時代 |
|---|---|
| 実装が正しいかチェックする | 何を作るかを決める |
| コーディング規約を守っているか見る | どんな規約にするかを決める |
| バグを見つける | バグが入れない仕組みを設計する |
| PR にコメントする | タスク開始前に acceptance criteria を書く |

「コードを書いた後に人間が見る」という前提自体が古くなっている。AI と CI と自動ロールバックの組み合わせは、**人間レビュアーより速く、疲れず、一貫している**。

## なぜ AWS アカウント構成が先か

「レビューなしでマージできる」ようにしたとき、最悪のケースは何か。

- バグのある Terraform が prod に適用されて本番インフラが壊れる
- セキュリティホールのある Lambda が prod にデプロイされる

これを許容可能なリスクにするには、**自動化されたゲートが壁になっていて、それを人間が操作では突破できない構造**が必要だ。アカウント構成がその壁を作る。

---

## AWS アカウント構成

```
AWS Organizations
├── Management Account（請求・SCP 管理のみ）
├── Security Account（CloudTrail・GuardDuty・Security Hub の集約）
├── dev Account
│   ├── 開発者: AdministratorAccess に近い権限
│   └── 手動デプロイ可（試行錯誤の場）
├── staging Account
│   ├── 開発者: ReadOnly のみ
│   └── デプロイ権限: CI/CD の IAM ロールのみ
└── prod Account
    ├── 開発者: ReadOnly のみ
    └── デプロイ権限: CI/CD の IAM ロールのみ
```

**ポイント：開発者は staging/prod に対して読み取りしかできない。**

デプロイは GitHub Actions が OIDC で assume role して実行する。このロールは CI/CD パイプラインにしか渡っていない。開発者が `aws lambda update-function-code` を手動で打っても、staging/prod では権限エラーになる。

### SCP でさらに強制する

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DenyDirectLambdaDeployOutsideCI",
      "Effect": "Deny",
      "Action": [
        "lambda:UpdateFunctionCode",
        "lambda:UpdateFunctionConfiguration"
      ],
      "Resource": "*",
      "Condition": {
        "StringNotLike": {
          "aws:PrincipalARN": "arn:aws:iam::*:role/github-actions-deploy-*"
        }
      }
    }
  ]
}
```

`github-actions-deploy-*` ロール以外からの Lambda 更新を SCP で拒否する。Terraform の `aws_lambda_function` も同様。

### GitHub Actions → AWS の OIDC 認証

```yaml
# .github/workflows/deploy.yml
permissions:
  id-token: write
  contents: read

jobs:
  deploy:
    steps:
      - name: AWS 認証（staging）
        uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: arn:aws:iam::STAGING_ACCOUNT_ID:role/github-actions-deploy-staging
          aws-region: ap-northeast-1
```

```hcl
# Terraform: OIDC Provider と IAM Role
resource "aws_iam_openid_connect_provider" "github" {
  url             = "https://token.actions.githubusercontent.com"
  client_id_list  = ["sts.amazonaws.com"]
  thumbprint_list = ["6938fd4d98bab03faadb97b34396831e3780aea1"]
}

resource "aws_iam_role" "github_actions_deploy" {
  name = "github-actions-deploy-staging"

  assume_role_policy = jsonencode({
    Version = "2012-10-17"
    Statement = [{
      Action = "sts:AssumeRoleWithWebIdentity"
      Effect = "Allow"
      Principal = {
        Federated = aws_iam_openid_connect_provider.github.arn
      }
      Condition = {
        StringLike = {
          "token.actions.githubusercontent.com:sub" = "repo:my-org/my-repo:ref:refs/heads/main"
        }
      }
    }]
  })
}
```

`main` ブランチからのみ assume role できる。PR ブランチからはできない。

## Terraform の自動検証パイプライン

### ツール構成

| ツール | 役割 | 何をブロックするか |
|---|---|---|
| `terraform fmt -check` | フォーマット | 書式が崩れたまま通らない |
| `terraform validate` | 構文チェック | 文法エラー・参照ミス |
| `tfsec` / `Checkov` | セキュリティ | S3 バケットが公開、暗号化なし等 |
| Conftest + OPA | カスタムポリシー | 組織固有のルール（タグ必須等）|
| `terraform test` | ユニットテスト | モジュールが期待通りの計画を出すか |

### GitHub Actions 設定例

```yaml
# .github/workflows/terraform.yml
name: Terraform CI/CD

on:
  pull_request:
    paths: ['terraform/**']
  push:
    branches: [main]
    paths: ['terraform/**']

jobs:
  validate:
    runs-on: ubuntu-latest
    defaults:
      run:
        working-directory: terraform/
    steps:
      - uses: actions/checkout@v4

      - uses: hashicorp/setup-terraform@v3

      - name: terraform fmt
        run: terraform fmt -check -recursive

      - name: terraform validate
        run: |
          terraform init -backend=false
          terraform validate

      - name: tfsec（セキュリティスキャン）
        uses: aquasecurity/tfsec-action@v1.0.0
        with:
          working_directory: terraform/

      - name: Checkov（ポリシーチェック）
        uses: bridgecrewio/checkov-action@v12
        with:
          directory: terraform/
          soft_fail: false  # Critical は必ずブロック

  plan:
    needs: validate
    runs-on: ubuntu-latest
    if: github.event_name == 'pull_request'
    steps:
      - uses: actions/checkout@v4

      - name: AWS 認証
        uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: ${{ secrets.STAGING_DEPLOY_ROLE }}
          aws-region: ap-northeast-1

      - name: terraform plan
        run: terraform plan -out=plan.tfplan

      # PR に plan 結果をコメントとして投稿
      - name: PR に plan 結果を投稿
        uses: actions/github-script@v7
        with:
          script: |
            const plan = require('fs').readFileSync('plan.txt', 'utf8');
            github.rest.issues.createComment({
              issue_number: context.issue.number,
              owner: context.repo.owner,
              repo: context.repo.repo,
              body: `## Terraform Plan\n\`\`\`\n${plan}\n\`\`\``
            });

  apply:
    needs: validate
    runs-on: ubuntu-latest
    if: github.ref == 'refs/heads/main' && github.event_name == 'push'
    steps:
      - uses: actions/checkout@v4

      - name: AWS 認証（staging）
        uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: ${{ secrets.STAGING_DEPLOY_ROLE }}
          aws-region: ap-northeast-1

      - name: terraform apply
        run: |
          terraform init
          terraform apply -auto-approve
```

PR 時は plan 結果がコメントとして自動投稿される。merge 後に自動 apply。**human が「LGTM」を押す必要がない。**

### Terraform ユニットテスト（terraform test）

Terraform 1.6 から built-in のテストフレームワークが使える。

```hcl

  command = plan

  assert {
    condition     = aws_lambda_function.this.runtime == "python3.12"

  assert {
    condition     = aws_lambda_function.this.reserved_concurrent_executions != -1
    error_message = "同時実行数の上限を必ず設定すること"
  }

run "encryption_enabled" {
  command = plan

  assert {
    condition     = aws_lambda_function.this.kms_key_arn != ""

  }

```

```bash
terraform test
```

これがパスしないと CI がブロックする。

## Python Lambda の自動検証パイプライン

### ツール構成

| ツール | 役割 | 設定の目安 |
|---|---|---|
| `pytest` + `pytest-cov` | テスト + カバレッジ | 閾値 80%（コアロジックは 90%+）|
| `mypy` | 型チェック | `strict` モードが理想 |
| `bandit` | Python SAST | high severity はブロック |
| `pip-audit` | 依存関係脆弱性 + Slopsquatting | critical はブロック |
| `ruff` | Lint + フォーマット | black + flake8 の代替（高速）|

### Lambda handler のテスト例

```python
# src/handler.py
import json
import boto3
from typing import Any

def handler(event: dict[str, Any], context: Any) -> dict[str, Any]:
    user_id = event.get("pathParameters", {}).get("userId")
    if not user_id:
        return {"statusCode": 400, "body": json.dumps({"error": "userId required"})}

    # DynamoDB から取得
    table = boto3.resource("dynamodb").Table("users")
    item = table.get_item(Key={"userId": user_id}).get("Item")

    if not item:
        return {"statusCode": 404, "body": json.dumps({"error": "Not found"})}

    return {"statusCode": 200, "body": json.dumps(item)}
```

```python
# tests/test_handler.py
import json
from unittest.mock import patch, MagicMock
import pytest
from src.handler import handler

@pytest.fixture
def mock_dynamodb_table():
    with patch("boto3.resource") as mock_resource:
        mock_table = MagicMock()
        mock_resource.return_value.Table.return_value = mock_table
        yield mock_table

def test_get_user_success(mock_dynamodb_table):
    mock_dynamodb_table.get_item.return_value = {
        "Item": {"userId": "user-123", "name": "Test User"}
    }
    event = {"pathParameters": {"userId": "user-123"}}
    response = handler(event, None)

    assert response["statusCode"] == 200
    body = json.loads(response["body"])
    assert body["userId"] == "user-123"

def test_get_user_not_found(mock_dynamodb_table):
    mock_dynamodb_table.get_item.return_value = {}
    event = {"pathParameters": {"userId": "nonexistent"}}
    response = handler(event, None)

    assert response["statusCode"] == 404

def test_missing_user_id():
    response = handler({"pathParameters": {}}, None)
    assert response["statusCode"] == 400
```

### GitHub Actions 設定例

```yaml
# .github/workflows/lambda.yml
name: Lambda CI/CD

on:
  pull_request:
    paths: ['src/**', 'tests/**', 'requirements*.txt']
  push:
    branches: [main]
    paths: ['src/**', 'tests/**', 'requirements*.txt']

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-python@v5
        with:
          python-version: '3.12'

      - name: 依存関係インストール
        run: pip install -r requirements-dev.txt

      - name: テスト + カバレッジ
        run: pytest --cov=src --cov-fail-under=80 --cov-report=xml

      - name: 型チェック
        run: mypy src/ --strict

      - name: Lint
        run: ruff check src/ tests/

      - name: Python SAST（bandit）
        run: bandit -r src/ -ll  # HIGH 以上のみブロック

      - name: 依存関係の脆弱性チェック（Slopsquatting 対策含む）
        run: pip-audit --requirement requirements.txt

  deploy:
    needs: test
    runs-on: ubuntu-latest
    if: github.ref == 'refs/heads/main' && github.event_name == 'push'
    steps:
      - uses: actions/checkout@v4

      - name: AWS 認証
        uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: ${{ secrets.STAGING_DEPLOY_ROLE }}
          aws-region: ap-northeast-1

      - name: Lambda パッケージ作成
        run: |
          pip install -r requirements.txt -t package/
          cp -r src/ package/
          cd package && zip -r ../lambda.zip .

      - name: Lambda デプロイ
        run: |
          aws lambda update-function-code \
            --function-name my-function \
            --zip-file fileb://lambda.zip \
            --publish  # 新しいバージョンを作成

      - name: エイリアスを新バージョンに向ける
        run: |
          VERSION=$(aws lambda list-versions-by-function \
            --function-name my-function \
            --query 'Versions[-1].Version' --output text)
          aws lambda update-alias \
            --function-name my-function \
            --name live \
            --function-version $VERSION
```

## GitHub の auto-merge 設定

### Branch Protection Rules

```
Settings > Branches > Branch protection rules > main

✓ Require status checks to pass before merging
  - Required checks:
    ✓ test（Lambda CI）
    ✓ validate（Terraform CI）
    ✓ deploy（staging へのデプロイ確認）

✗ Require a pull request before merging
  → レビュアーの承認を必須にしない

✓ Allow auto-merge
```

### PR の自動マージ設定

```yaml
# .github/workflows/automerge.yml
name: Auto Merge

on:
  workflow_run:
    workflows: ['Lambda CI/CD', 'Terraform CI/CD']
    types: [completed]

jobs:
  automerge:
    runs-on: ubuntu-latest
    if: github.event.workflow_run.conclusion == 'success'
    steps:
      - name: Auto merge
        uses: pascalgn/automerge-action@v0.16.3
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
          MERGE_METHOD: squash
          MERGE_COMMIT_MESSAGE: pull-request-title
```

全チェックが green になった PR は自動的に squash merge される。

## Lambda の自動ロールバック

本番で問題が起きたとき、人間が気づいて操作するまでの間をなくす。

### CloudWatch アラーム + Lambda エイリアス切り替え

```hcl
# Terraform でアラームとロールバックを管理

resource "aws_lambda_alias" "live" {
  name             = "live"
  function_name    = aws_lambda_function.this.function_name
  function_version = aws_lambda_function.this.version
}

# エラーレートが 5% を超えたらアラーム
resource "aws_cloudwatch_metric_alarm" "lambda_error_rate" {
  alarm_name          = "lambda-error-rate-high"
  comparison_operator = "GreaterThanThreshold"
  evaluation_periods  = 2
  metric_name         = "Errors"
  namespace           = "AWS/Lambda"
  period              = 60
  statistic           = "Sum"
  threshold           = 5

  dimensions = {
    FunctionName = aws_lambda_function.this.function_name
    Resource     = "${aws_lambda_function.this.function_name}:live"
  }

  alarm_actions = [aws_sns_topic.rollback.arn]
}

# アラーム → SNS → Lambda でロールバック
resource "aws_sns_topic_subscription" "rollback" {
  topic_arn = aws_sns_topic.rollback.arn
  protocol  = "lambda"
  endpoint  = aws_lambda_function.rollback.arn
}
```

```python
# ロールバック用 Lambda
import boto3

def handler(event, context):
    lambda_client = boto3.client("lambda")
    function_name = "my-function"

    # 現在の live エイリアスのバージョンを確認
    alias = lambda_client.get_alias(FunctionName=function_name, Name="live")
    current_version = int(alias["FunctionVersion"])

    if current_version <= 1:
        print("ロールバック先がない")
        return

    # 1つ前のバージョンに戻す
    previous_version = str(current_version - 1)
    lambda_client.update_alias(
        FunctionName=function_name,
        Name="live",
        FunctionVersion=previous_version
    )
    print(f"ロールバック完了: {current_version} → {previous_version}")
```

エラーレートが閾値を超えたら、人間が何もしなくても前バージョンに戻る。

## フィーチャーフラグとの組み合わせ（高リスク変更向け）

自動検証を通ったとしても、ビジネスロジックの大きな変更は段階的に公開したい。

```python
# AppConfig でフラグを取得
import boto3
import json

def get_flags():
    client = boto3.client("appconfigdata")
    session = client.start_configuration_session(
        ApplicationIdentifier="my-app",
        EnvironmentIdentifier="production",
        ConfigurationProfileIdentifier="feature-flags",
        RequiredMinimumPollIntervalInSeconds=30,
    )
    config = client.get_latest_configuration(
        ConfigurationToken=session["InitialConfigurationToken"]
    )
    return json.loads(config["Configuration"].read())

def handler(event, context):
    flags = get_flags()
    if flags.get("new_pricing_logic"):
        return new_pricing(event)
    return old_pricing(event)
```

「コードレビューなしでマージ・デプロイ」＋「フラグで段階的に公開」を組み合わせると、人間が介在しなくても安全にリリースできる。

## どこから始めるか：最初の「意外とイケる」体験を作る

「全部いきなりやめる」必要はない。最初は**壊れても誰も困らないものだけ**でいい。

### エリアの分類

| 影響範囲 | 具体例 | 戦略 |
|---|---|---|
| 小：壊れても誰も困らない | 社内向けバッチ、定期レポート生成 Lambda、開発ツール | まずここだけレビューなし・自動マージにする |
| 中：影響はあるが限定的 | フィーチャーフラグの裏側、社内ダッシュボード、staging 環境のみ | 体験を積んだら移行 |
| 大：本番 core に直撃 | ユーザー向け API、決済、認証 | 自動化ゲートが実績を積んでから |

### 「壊れても困らない Lambda」で試す

たとえば「毎週月曜に DynamoDB を集計して Slack に送る Lambda」があるとする。

- バグが出た最悪のケース：集計が間違った数字を Slack に投げる
- 自動ロールバックが動いた場合：次のデプロイまで前週と同じ集計結果が届く

この Lambda の PR レビューに週 30 分かけていたとしたら、1年で 26 時間。得られていた安全性と比べてコストが見合っているか？

「この Lambda はレビューなしにしてみよう」と決めて 1 ヶ月運用する。CI が毎回 green で、Slack の集計も変わらず届いて、何も起きない。**「あ、これ普通に大丈夫だ」という体感が生まれる。**

その体感がないまま「設計上は安全なはず」と理屈だけで core を移行するのはリスクが高い。体感を積み上げてから判断する。

## 段階的に進めるロードマップ

### Phase 1：自動化ゲートを整備する（1〜2 ヶ月）

- [ ] pytest + coverage 閾値 80%
- [ ] mypy + ruff を CI に追加
- [ ] bandit + pip-audit を CI に追加
- [ ] tfsec + Checkov を CI に追加
- [ ] terraform validate + fmt -check を CI に追加
- 現時点ではまだレビューを続ける。自動化が「常に通る」ことを確認する期間

### Phase 2：AWS アカウント構成を整える（1〜2 ヶ月）

- [ ] staging / prod アカウントの開発者権限を ReadOnly に絞る
- [ ] GitHub Actions OIDC の設定
- [ ] SCP で直接デプロイを禁止
- [ ] Lambda エイリアス + 自動ロールバックの設定

### Phase 3：影響が小さいエリアから auto-merge を有効にする

- [ ] まず「壊れても困らないリポジトリ or ブランチ」に限定して有効にする
- [ ] Branch protection rules でレビュアー承認を必須から外す
- [ ] Required status checks を設定
- [ ] auto-merge を有効にする
- [ ] 1〜2 週間観察。自動ロールバックが効いているか確認
- [ ] 問題なければ影響範囲「中」のエリアに広げる

### Phase 4：フィーチャーフラグを導入する（高リスク変更向け）

- [ ] AWS AppConfig でフラグ管理
- [ ] カナリアデプロイ（Lambda Alias の加重ルーティング）を設定

### どうしても残すレビュー

- **DB マイグレーション**：スキーマ変更は自動ロールバックが効きにくい。マイグレーション PR だけ手動レビューを残す。
- **IAM・SCP の変更**：間違えると全体に影響する。Terraform Cloud の Sentinel などでポリシー強制し、それでもカバーできない変更は手動レビュー。
- **クロスサービスの破壊的変更**：コントラクトテストで自動化できる部分は自動化する。
