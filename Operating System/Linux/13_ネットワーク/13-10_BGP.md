# BGP

## BGP が必要になる場面

オンプレのデータセンターと AWS を Direct Connect で繋いだとき、こういう問題が出る。

「AWS 側で新しいサブネットを作った。オンプレ側からそこに通信させたい。でも毎回オンプレのルータに手動でルートを追加するのはつらい」

逆もある。「オンプレ側に新しいサーバーを立てた。AWS 側からそこへ通信させたい」

手動でルートを管理すると、追加・削除のたびに両側の設定変更が必要で、ミスや漏れが起きる。BGP はこれを自動化する仕組みだ。

---

## 核心：「AS」と「ピア」

BGP を理解するのに必要な概念は2つだけ。

### AS（Autonomous System、自律システム）

ネットワークの「管轄単位」。

- 自社のオンプレネットワーク全体 → ひとつの AS
- AWS の東京リージョン → AWS が管理する AS
- ISP（インターネットプロバイダ）→ 各社がそれぞれ AS を持つ

各 AS には **AS 番号（ASN）** が割り振られる。グローバルで一意な番号（パブリック AS 番号）と、自社内や閉じた環境だけで使うプライベート AS 番号（64512〜65534）がある。

### ピア（BGP ネイバー）

「このルーターと BGP でルート情報を交換する」と決めた相手のこと。BGP はピアと1対1でセッションを張り、お互いに「うちの AS ではこのアドレス帯を持っている」を伝え合う。

## 運用担当が実際にやること

### 1. ASN を決める

自社ネットワークに番号をつける。閉じた環境（オンプレ↔AWS など）ならプライベート ASN（64512〜65534）で十分。インターネット上でルーティングしたい場合はパブリック ASN が必要（JPNIC 等に申請）。

```
オンプレ: AS 65001
AWS（VGW）: AS 64512（デフォルト）
```

### 2. ピアの設定

「誰と話すか」を設定する。必要な情報は3つ：

| 設定項目 | 内容 |
|---|---|
| ピアの IP アドレス | 相手ルーターの IP（Direct Connect なら AWS 側の BGP IP）|
| ピアの ASN | 相手の AS 番号 |
| 認証パスワード（MD5） | セッションの正当性確認。省略可だが推奨 |

```
# Cisco IOS の例（イメージ）
router bgp 65001
  neighbor 169.254.0.1 remote-as 64512
  neighbor 169.254.0.1 password MySecret
```

### 3. 広告するプレフィックスを決める

「うちの AS はこのアドレス帯を持っている」と相手に教えるリストを作る。

```
# 自社オンプレが 10.0.0.0/8 を持っていると広告する
router bgp 65001
  network 10.0.0.0 mask 255.0.0.0
```

**重要**：広告するプレフィックスは絞ること。余計なものを広告すると意図しないルートが相手に伝わる。

### 4. 受け入れるルートをフィルタリングする

ピアから送られてくるルートをすべて受け入れるのは危険。プレフィックスリストやルートマップで「これだけ受け入れる」と明示する。

```
# 172.31.0.0/16 だけ受け入れる（AWS VPC からのルート）
ip prefix-list FROM-AWS permit 172.31.0.0/16
ip prefix-list FROM-AWS deny 0.0.0.0/0 le 32

router bgp 65001
  neighbor 169.254.0.1 prefix-list FROM-AWS in
```

### 5. BGP セッションの監視

BGP セッションが落ちると「片側からだけ通信できる」「どこにも届かない」という障害になる。監視対象は主に2つ：

- **BGP セッション状態**：`Established`（正常）かどうか
- **受信・送信ルート数**：期待する数のルートが来ているか

```bash
# Cisco IOS でのセッション確認
show bgp summary

# 受信ルートの確認
show bgp neighbors 169.254.0.1 received-routes
```

## AWS での登場シーン

### Direct Connect

オンプレ↔AWS 間をBGPで接続する代表例。AWS 側は Virtual Private Gateway（VGW）または Direct Connect Gateway が BGP ピアになる。

```
オンプレルーター ──BGP── AWS VGW
  AS 65001              AS 64512

オンプレが広告: 10.0.0.0/8
AWS が広告:    172.31.0.0/16（VPC の CIDR）
```

Direct Connect を設定する運用担当がやることは：
1. AWS コンソールで VIF（Virtual Interface）を作成 → AWS 側の BGP IP と ASN が払い出される
2. その情報をオンプレのルーターに入力してピア設定
3. 広告プレフィックスとフィルターを設定
4. BGP セッションが `Established` になるまで待つ

### Site-to-Site VPN（動的ルーティング）

VPN でも BGP を使える（静的ルーティングの代わりに）。オンプレ側のルーターが BGP に対応していれば、VPC のサブネット追加が自動的にオンプレ側に伝わる。

### Transit Gateway

複数の VPC やオンプレを繋ぐハブ。オンプレとのBGP接続もTGW側で集約できる。

## よくある問題

| 症状 | 原因の見当をつける場所 |
|---|---|
| BGP セッションが `Established` にならない | IP アドレスミス、ASN ミス、MD5 パスワード不一致、ファイアウォールで TCP 179 が通っていない |
| ルートが受信できていない | フィルター設定の誤り、相手側が広告していない |
| 通信できるが片方向だけ | ルートが非対称（一方にしか経路が入っていない）|
| BGP は生きているが通信できない | ルートが入っているが VPC のルートテーブルに反映されていない |

---

## まとめ

BGP は「ネットワーク同士がルート情報を自動で交換する仕組み」で、手動ルート管理の限界を解決するために使う。

運用担当がやることを一言で言うと：
1. **ASN を決めてピアを設定する**（誰と話すか）
2. **広告するルートを定義する**（何を教えるか）
3. **受け入れるルートをフィルタリングする**（何を受け取るか）
4. **セッションとルート数を監視する**（ちゃんと動いているか）

この4つ。
