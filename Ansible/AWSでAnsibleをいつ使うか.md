# AWS で Ansible をいつ使うか

## 問い

AWS をメインに使っている現場で、「Ansible じゃないとダメ」なシーンは今どのくらいあるか。

結論から言うと、**フル AWS ネイティブ構成なら Ansible はほぼ不要**。残るのはハイブリッド環境とレガシーなステートフル EC2 がメインになってきた。

---

## 「Ansible っぽい用途」を AWS ネイティブで代替できる

Ansible を使いたくなる場面と、AWS ネイティブの代替手段の対応関係：

| やりたいこと | AWS ネイティブの代替 | 補足 |
|---|---|---|
| EC2 の初期セットアップ | Packer で AMI を焼く + Terraform | 設定変更 = 新 AMI に更新。Ansible の出番がなくなる |
| パッケージのインストール・更新 | SSM Patch Manager | スケジューリング・承認フロー付きで管理できる |
| 設定ファイルの配布・管理 | SSM State Manager | 望ましい状態を定義して継続的に適用できる |
| コマンド一括実行 | SSM Run Command | 複数インスタンスへの一括実行が GUI/CLI どちらでも可能 |
| ローリングデプロイ | CodeDeploy / ECS rolling update | EC2 でもコンテナでも対応 |

アプリがコンテナ（ECS/EKS）で動いていれば、そもそもサーバーの設定管理自体が不要になる。

---

## それでも Ansible が残るケース

### ① ハイブリッド環境（AWS + オンプレ混在）

**これが現時点で一番強い使い所。**

SSM も CodeDeploy も AWS インスタンスにしか届かない。オンプレのサーバーや他クラウドのインスタンスも同じ Playbook で一括管理したい場合、Ansible のエージェントレス（SSH）という特性が活きる。

```
[Ansible 実行マシン]
  |
  ├── SSH --> AWS EC2 (ap-northeast-1)
  ├── SSH --> オンプレ物理サーバー
  └── SSH --> 別クラウドの VM
```

AWS ツールだと横断できないが、Ansible なら SSH が通る場所はすべて対象にできる。

### ② 長寿命のステートフル EC2

イミュータブルにできない EC2 が残っている場合：

- EC2 上でセルフホストしている DB（RDS に移行できていない）
- 複雑な依存関係があって AMI を焼き直すたびに問題が起きるレガシーアプリ
- ライセンスや契約の都合でインスタンスを置き換えられない

こういったサーバーへの継続的な設定変更・ドリフト修正は Ansible が向いている。ただしこのパターン自体を減らしていく方向が正しい。

### ③ SSM が使えない制約がある場合

- インターネットアクセスもVPCエンドポイントも設定されていなくて SSM Agent が機能しない
- AWS 以外の環境と統一ツールにしたい運用上の制約

---

## 「Ansible を使わなくていい」判断基準

以下がすべて当てはまるなら、Ansible を新規導入する必要はほぼない：

- [ ] アプリはコンテナ（ECS/EKS）または Serverless（Lambda）で動いている
- [ ] EC2 を使う場合はイミュータブル運用（Packer + AMI 置き換え）ができる
- [ ] 管理対象は AWS インスタンスのみ（オンプレなし）
- [ ] SSM Agent が有効で、SSM Patch Manager / State Manager が使える

---

## Ansible の学習優先度

| 現場の状況 | 優先度 |
|---|---|
| オンプレ・ハイブリッド環境がある | 高い。今でも中心的なツール |
| レガシー EC2 が多くイミュータブル化できていない | 高い |
| フル AWS ネイティブ + コンテナ中心 | 低い。他に優先すべきことがある |
| これから新規構築する AWS 環境 | 低い。最初からイミュータブル設計にしたほうがいい |

Ansible 自体のスキルは汎用性が高いので、インフラエンジニアとして持っていて損はない。ただし「AWS でのインフラ管理を効率化したい」という目的なら、SSM や Packer を先に押さえるほうが現代的な AWS 環境では直接役に立つ。
