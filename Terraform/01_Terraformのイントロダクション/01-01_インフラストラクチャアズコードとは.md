# インフラストラクチャアズコード（IaC）とは

「本番と同じ環境を再現してほしい」と言われたとき、手順書を見ながらコンソールをポチポチすれば作れる。でもその手順が古かったり、人によって手順が違ったり、どの設定が最新なのかわからなくなる。本番と dev でいつの間にか構成が違う、という状況も起きる。

IaC はインフラの構成をコードで定義することで、こういった問題を解決する。

---

## コードで管理することで何が変わるか

| 手動管理 | IaC |
|---|---|
| コンソールを操作して設定 | コードを書いて apply |
| 「誰かが変えた」がわからない | Git の履歴で変更者・変更内容が追える |
| 同じ環境の再現に職人技が必要 | コードを apply すれば誰でも同じ環境が作れる |
| レビューできない | Pull Request でインフラ変更をレビューできる |

---

## 主な IaC ツール

| ツール | 特徴 |
|---|---|
| **Terraform** | クラウド非依存、HCL で記述、最も普及 |
| AWS CloudFormation | AWS 専用、YAML/JSON |
| Pulumi | TypeScript・Python など汎用言語で書ける |
| Ansible | サーバーの設定管理寄り |

複数クラウドを使う・クラウドに依存しないコードを書きたいなら Terraform が最有力候補。

---

## IaC が「ないとつらい」具体的な場面

- 障害対応でインフラを一から作り直す必要が出た
- dev と prod の構成差異でバグが出る
- 新メンバーが「どうやって環境作るの？」と詰まる
- 「このセキュリティグループ誰が何のために作ったの？」がわからない

これらが日常的に起きているなら IaC 導入の効果が高い。
