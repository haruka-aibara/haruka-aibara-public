# リソースの挙動

`terraform apply` を実行したとき、Terraform は「コードの定義」と「現在の State」を比べて何をすべきか判断する。この挙動を知らないと、意図しないリソースの削除や再作成が起きたときに焦る。

---

## 4 パターンの動作

| 操作 | いつ起きるか |
|---|---|
| **Create** | .tf にあるのに State にない → 新規作成 |
| **Read** | plan 実行時に現在の状態を API で取得 |
| **Update** | State と .tf に差分がある、かつ in-place で変更できる |
| **Delete** | State にあるのに .tf から消えた → 削除 |

---

## 一番注意が必要なのは Replace（再作成）

変更の種類によっては in-place 更新できず、**削除してから再作成**になる。`plan` の出力で `-/+` と表示される。

```
# aws_instance.web must be replaced
-/+  resource "aws_instance" "web" {
       ami = "ami-old" -> "ami-new"  # forces replacement
     }
```

EC2 の AMI を変えるとインスタンスが一度消える。本番環境で気づかずに apply するとダウンタイムが発生する。**apply 前に plan を必ず読む**習慣が大事。

---

## resource ブロックの書き方

```hcl
resource "aws_instance" "web" {
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = "t3.micro"

  tags = {
    Name = "web-server"
  }
}
```

`resource "リソースタイプ" "ローカル名"` の形式。`aws_instance.web` のように他のリソースから参照できる。

---

## リソース間の参照と依存解決

```hcl
resource "aws_eip" "web" {
  instance = aws_instance.web.id  # web の id を参照
}
```

`aws_instance.web.id` のように参照を書くと、Terraform は自動で依存関係を解析して `aws_instance` を先に作ってから `aws_eip` を作る。`depends_on` を明示的に書かなくていい。
