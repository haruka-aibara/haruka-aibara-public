# プロバイダーのバージョン管理

バージョンを固定しないと、`terraform init` のたびにプロバイダーが最新版になり、ある日突然 apply が壊れる。チームで使っているなら特に問題になりやすい。「自分のローカルでは動くのに CI では失敗する」の原因の一つ。

---

## バージョン制約の書き方

```hcl
terraform {
  required_version = "~> 1.7"   # Terraform 本体のバージョン

  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"        # プロバイダーのバージョン
    }
  }
}
```

---

## 演算子の意味

| 演算子 | 意味 | 例 |
|---|---|---|
| `= 5.31.0` | 完全固定 | `5.31.0` のみ |
| `>= 5.0` | 以上 | `5.0.0` 以上すべて |
| `~> 5.0` | パッチのみ更新許可 | `5.0.x`（`5.1` は NG） |
| `~> 5` | マイナーまで更新許可 | `5.x.x`（`6` は NG） |

実務では `~>` が多い。メジャーバージョンの破壊的変更を防ぎつつ、バグ修正は取り込める。

---

## `terraform.lock.hcl` との関係

`terraform init` を実行すると、制約を満たす実際のバージョンが `terraform.lock.hcl` に記録される。

```hcl
provider "registry.terraform.io/hashicorp/aws" {
  version     = "5.31.0"
  constraints = "~> 5.0"
  hashes = ["h1:..."]
}
```

このファイルを Git にコミットすることで、全員・CI が同じバージョンを使う。

---

## アップデートするとき

```bash
# 制約の範囲内の最新バージョンに更新
terraform init -upgrade

# lock.hcl の差分を確認してコミット
git diff .terraform.lock.hcl
git add .terraform.lock.hcl
```

メジャーバージョンを上げるときは必ず Changelog と移行ガイドを確認する。
