# HCP Terraform をいつ使うか

「S3 バックエンドを自分で用意して、DynamoDB でロックして、GitHub Actions でワークフローを組んで——」これを全部自前でやるのが面倒に感じたら HCP Terraform（旧 Terraform Cloud）の出番。State・実行環境・承認フロー・ポリシーチェックをまとめて提供するマネージドサービス。

---

## HCP Terraform が解決する課題

| 課題 | HCP Terraform の解決策 |
|---|---|
| State をどこに保存するか | 内蔵の State バックエンドを提供 |
| plan / apply の実行環境 | クラウドのエフェメラル環境で実行 |
| シークレットの管理 | Workspace Variables として暗号化保存 |
| チームのアクセス制御 | ロールベースアクセス制御（RBAC） |
| ポリシー適用 | Sentinel ポリシーエンジン |

---

## 使うべき状況

### ✅ 向いているケース

- チームで Terraform を使っていて、アクセス制御・承認フローが必要
- S3 バックエンドや State ロックの設定が面倒
- Sentinel によるコンプライアンス強制が必要
- セルフホストの CI を持っていない（GitHub Actions 代わりに使う）

### ❌ 向いていないケース

- 個人・小規模チームで S3 バックエンドで十分
- 既存の GitHub Actions / GitLab CI でワークフローが確立している
- オンプレ環境で外部サービスへの接続が制限されている

## プランと費用

| プラン | 特徴 | 費用 |
|---|---|---|
| Free | 500リソースまで。1ワークスペース | 無料 |
| Plus | 無制限リソース。SSO・監査ログ | $20/ユーザー/月 |
| Enterprise | Terraform Enterprise（セルフホスト版） | 要問い合わせ |

小規模チームなら Free プランで十分使える。

---

## GitHub Actions との比較

```
# GitHub Actions でやること
- S3 バックエンドを自前で用意
- State ロックの設定（DynamoDB）
- 認証情報の管理（Actions Secrets）
- plan/apply のワークフロー実装

# HCP Terraform でやること
- ワークスペースを作るだけ
```

セットアップコストは HCP Terraform の方が低い。カスタマイズ性は GitHub Actions の方が高い。

---

## 接続方法

```hcl
terraform {
  cloud {
    organization = "my-org"
    workspaces {
      name = "production"
    }
  }
}
```

`terraform login` でトークンを取得すれば `terraform init` だけで設定完了。
