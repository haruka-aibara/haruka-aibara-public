# テストモジュール

`terraform test` でテスト専用のモジュールを使って、ヘルパー処理や外部検証をモジュール化する手法。

---

## テストモジュールとは

`.tftest.hcl` の `run` ブロック内で `module` ブロックを使うことで、テスト用の Terraform モジュールを呼び出せる。

```hcl
run "verify_endpoint" {
  module {
    source = "./tests/modules/http_check"
  }

  variables {
    url = "http://${run.deploy.alb_dns_name}/health"
  }

  assert {
    condition     = module.http_check.status_code == 200
    error_message = "ヘルスチェックが失敗しています"
  }
}
```

---

## テストモジュールの例

`tests/modules/http_check/main.tf`:

```hcl
variable "url" {
  type = string
}

data "http" "check" {
  url = var.url
}

output "status_code" {
  value = data.http.check.status_code
}

output "body" {
  value = data.http.check.response_body
}
```

`data "http"` でHTTP リクエストを送り、ステータスコードを返すシンプルなモジュール。

## モックプロバイダー

Terraform 1.7 以降、テスト内でプロバイダーをモックできる。実際のクラウドリソースを作らずにテストが完結する。

```hcl
# tests/unit.tftest.hcl

mock_provider "aws" {
  mock_resource "aws_vpc" {
    defaults = {
      id         = "vpc-mock123"
      cidr_block = "10.0.0.0/16"
    }
  }
}

run "verify_vpc" {
  command = plan

  assert {
    condition     = aws_vpc.main.cidr_block == "10.0.0.0/16"
    error_message = "VPC の CIDR が想定と違います"
  }
}
```

## ディレクトリ構成のベストプラクティス

```
modules/
└── networking/
    ├── main.tf
    ├── variables.tf
    ├── outputs.tf
    └── tests/
        ├── unit.tftest.hcl        # plan のみ・高速
        ├── integration.tftest.hcl # apply・実リソース作成
        └── modules/
            └── http_check/        # テストヘルパーモジュール
                ├── main.tf
                └── outputs.tf
```

テスト用モジュールをメインのモジュールと分離しておくと再利用しやすい。

---

## `terraform test` の実行オプション

```bash
# 全テスト実行
terraform test

# 特定のファイルだけ実行
terraform test -filter=tests/unit.tftest.hcl

# verbose 出力
terraform test -verbose
```
