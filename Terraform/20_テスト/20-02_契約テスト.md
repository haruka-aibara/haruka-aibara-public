# 契約テスト

契約テスト（Contract Testing）は、モジュールが公開する入出力インターフェースが期待どおりであることを検証するテスト。モジュールの利用者と提供者の「契約」を守るための仕組み。

---

## 何をテストするか

- 入力変数のバリデーションが正しく機能するか
- 出力値が期待する型・形式になっているか
- モジュールの動作保証（インターフェース）を文書化する

---

## terraform test による契約テスト

```hcl
# tests/contract.tftest.hcl

# 正常系：正しい入力値
run "valid_inputs" {
  command = plan

  variables {
    instance_type = "t3.micro"
    environment   = "production"
  }

  # outputs が期待の型になっているか
  assert {
    condition     = output.instance_id != ""
    error_message = "instance_id が空です"
  }
}

# 異常系：不正な入力値でエラーになるか
run "invalid_environment" {
  command = plan

  variables {
    instance_type = "t3.micro"
    environment   = "invalid"   # 許可されていない値
  }

  expect_failures = [
    var.environment,
  ]
}
```

---

## 入力変数のバリデーション定義

契約テストと対になるのが変数側の `validation` ブロック。

```hcl
variable "environment" {
  type = string

  validation {
    condition     = contains(["development", "staging", "production"], var.environment)
    error_message = "environment は development / staging / production のいずれかです"
  }
}
```

`expect_failures` でこのバリデーションエラーが発生することをテストできる。

---

## 実務での使い所

複数チームが共通モジュールを使う環境で有効。

- チーム A がモジュールを変更したとき、既存の入出力インターフェースが壊れていないか確認できる
- 変数名の変更や出力の削除が他チームへの影響を早期検出できる

モジュールのリファクタリング時に特に重宝する。
