# 事前条件（precondition）

「本番の LB は削除保護が有効のはず」「AMI は x86_64 のはず」—こういった前提条件をコードで表明できる。条件を満たさないまま apply しようとするとエラーになり、意図しない状態のデプロイを防げる。

---

## output の precondition

出力値を返す前に条件を検証する。

```hcl
output "load_balancer_dns" {
  value = aws_lb.main.dns_name

  precondition {
    condition     = aws_lb.main.enable_deletion_protection
    error_message = "本番 LB は削除保護を有効にしてください。"
  }
}
```

削除保護が無効のまま apply しようとするとエラーになる。

---

## resource の precondition / postcondition

`resource` ブロックの `lifecycle` 内にも書ける（Terraform v1.2+）。

```hcl
resource "aws_instance" "web" {
  ami           = data.aws_ami.app.id
  instance_type = var.instance_type

  lifecycle {
    # apply 前に評価：AMI のアーキテクチャが合っているか
    precondition {
      condition     = data.aws_ami.app.architecture == "x86_64"
      error_message = "AMI は x86_64 アーキテクチャを使ってください。"
    }

    # apply 後に評価：パブリック IP が割り当てられたか
    postcondition {
      condition     = self.public_ip != ""
      error_message = "インスタンスにパブリック IP が割り当てられませんでした。"
    }
  }
}
```

---

## variable の validation との違い

| | `variable validation` | `precondition / postcondition` |
|---|---|---|
| 評価タイミング | plan 前（変数入力時） | plan / apply 時 |
| 参照できる値 | `var.変数名` のみ | リソース属性・data ソースも参照可 |
| 向いている用途 | 入力値の範囲チェック | インフラの実際の状態の検証 |

「t3.micro か t3.small だけ許可する」は validation で。「本番環境のリソースは削除保護が必須」は precondition で書く。
