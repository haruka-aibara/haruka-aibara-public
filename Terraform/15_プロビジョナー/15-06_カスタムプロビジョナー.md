# カスタムプロビジョナー

Terraform には組み込みプロビジョナー（`file`・`local-exec`・`remote-exec`）以外に、サードパーティ製のカスタムプロビジョナーを使う方法がある。

---

## なぜカスタムプロビジョナーが必要か

組み込みプロビジョナーでは対応できないケース：

- Puppet や Chef など特定の構成管理ツールとの統合
- 独自のデプロイシステムへの組み込み
- 組み込みプロビジョナーより高機能な接続・転送が必要な場合

---

## サードパーティ製プロビジョナーの例

### chef プロビジョナー

```hcl
resource "aws_instance" "web" {
  # ...

  provisioner "chef" {
    server_url      = "https://chef.example.com/organizations/my_org"
    node_name       = "web-${count.index}"
    run_list        = ["recipe[apache]"]
    user_name       = "deploy"
    user_key        = file("~/.chef/deploy.pem")
    client_options  = ["chef_license 'accept'"]
  }
}
```

※ `chef` プロビジョナーは Terraform 0.13 以降、公式から削除されたため外部プラグインとして提供されている。

---

## カスタムプロビジョナーのインストール方法

Terraform プロビジョナーはプラグイン形式。バイナリを `~/.terraform.d/plugins/` に配置する。

```bash
# macOS の例
cp terraform-provisioner-custom ~/.terraform.d/plugins/terraform-provisioner-custom_v1.0.0
chmod +x ~/.terraform.d/plugins/terraform-provisioner-custom_v1.0.0
```

---

## 実務での方針

カスタムプロビジョナーはメンテナンスコストが高い。

| アプローチ | 採用基準 |
|---|---|
| User Data / cloud-init | EC2 の初期設定（最優先） |
| AWS Systems Manager | エージェント経由で安全に実行したい |
| `local-exec` + curl | 外部 API 連携（軽量） |
| カスタムプロビジョナー | 上記で対応できない場合のみ |

公式の推奨は「プロビジョナーを使わない設計を先に検討する」こと。カスタムプロビジョナーはその中でも最後の手段。
