# プロビジョナーを使うタイミング

プロビジョナーは**最後の手段**として使う機能。可能な限り使わない設計にすることが推奨されている。

---

## HashiCorp の公式見解

> Provisioners should only be used as a last resort. For most common situations there are better alternatives.

プロビジョナーは以下の問題を持つ：

- 実行結果が State に記録されない（冪等性がない）
- ネットワーク接続・SSH アクセスが必要で CI/CD で設定が複雑
- エラー時の挙動が予測しにくい

---

## 代替手段（推奨）

| やりたいこと | プロビジョナーの代替 |
|---|---|
| EC2 の初期設定 | User Data スクリプト または AMI をベイク |
| アプリのデプロイ | CodeDeploy / ECS / Lambda |
| 設定ファイルの配置 | AWS Systems Manager（SSM）|
| ファイル転送 | S3 + CloudFormation Init |

---

## 使う場面

以下のような、どうしても外部コマンドを実行しなければならない場合に限定する：

- Terraform プロバイダーが存在しない独自システムへの登録
- リソース作成後に外部 API を叩く必要がある
- ローカルで後処理スクリプトを実行したい（`local-exec`）

---

## プロビジョナーが必要な場合の方針

1. `local-exec` で済む処理は `local-exec` を使う（SSH 不要）
2. `null_resource` と組み合わせてリソースの作成に紐づける
3. 冪等にならない処理であることを明示的にコメントに残す
