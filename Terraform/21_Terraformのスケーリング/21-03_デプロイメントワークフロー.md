# デプロイメントワークフロー

チームで Terraform を運用するときの標準的なワークフロー。レビューと承認を挟むことで誤った変更を防ぐ。

---

## 基本フロー（PR ベース）

```
1. ブランチを切って変更
2. PR を作成 → CI で terraform plan を自動実行
3. plan 結果を PR にコメント
4. レビュー・承認
5. main にマージ → terraform apply を自動実行
```

---

## Atlantis

PR ベースの Terraform ワークフローを自動化する OSS ツール。

```
# PR にコメントするだけで操作できる
atlantis plan    # plan 実行 → 結果を PR にコメント
atlantis apply   # 承認後に apply
```

GitHub / GitLab / Bitbucket に対応。セルフホストが必要。

### 主な機能

- PR ごとに自動 plan
- `atlantis apply` コマンドで承認フロー
- ロック機能（並行した apply を防止）
- ポリシーチェックとの統合

---

## Terraform Cloud / HCP Terraform

HashiCorp が提供する管理型ワークフロー。

```hcl
terraform {
  cloud {
    organization = "my-org"
    workspaces {
      name = "production"
    }
  }
}
```

VCS（GitHub など）と連携し、PR に plan 結果を自動コメント。apply には UI での承認フローがある。

---

## ワークフローの原則

| 原則 | 内容 |
|---|---|
| PR なしで apply しない | main への直接 push を禁止 |
| plan を必ずレビューする | apply 前に差分確認を義務付け |
| apply は自動化する | 承認後は人手を介さず CI が apply |
| plan と apply は同じ artifact | plan 後に変更が入ると apply が別内容になる |

plan の結果ファイル（tfplan）を artifact として保存し、apply はその artifact を使うことで「レビューした plan と異なる apply」を防ぐ。

```bash
terraform plan -out=tfplan
# ↑ これを artifact に保存

terraform apply tfplan
# ↑ artifact を使って apply
```
