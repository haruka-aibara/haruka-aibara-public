# Kubernetes トラブルシューティング

「Pod が CrashLoopBackOff になっている」「デプロイしたのにトラフィックが来ない」——Kubernetes の障害調査はコマンドを組み合わせて原因を絞り込む。よくあるパターンと調査手順をまとめる。

---

## 調査の基本フロー

```
kubectl get pods        # 状態確認
  ↓ 異常があれば
kubectl describe pod <name>  # イベントとステータス確認
  ↓ コンテナが起動していれば
kubectl logs <pod>      # ログ確認
  ↓ ネットワーク問題なら
kubectl exec -it <pod> -- curl http://service:port  # 疎通確認
```

---

## CrashLoopBackOff

コンテナが起動しては落ちることを繰り返している。

```bash
# ログを確認（前のコンテナのログ）
kubectl logs <pod> --previous

# 詳細なイベントを確認
kubectl describe pod <pod>
# → Events セクションの "Back-off restarting failed container" など

# よくある原因:
# 1. アプリの起動時エラー（DB 接続失敗・設定ミス）
# 2. OOMKilled（メモリ不足で強制終了）
# 3. コマンドの typo（Dockerfile の CMD ミス）
```

**OOMKilled の場合:**

```bash
kubectl describe pod <pod>
# Reason: OOMKilled → resources.limits.memory を増やす

# または requests/limits の設定を確認
kubectl get pod <pod> -o jsonpath='{.spec.containers[*].resources}'
```

## Pending 状態から動かない

Pod がスケジュールされていない。

```bash
kubectl describe pod <pod>
# Events を確認

# よくある原因と対処:
# "Insufficient cpu" / "Insufficient memory"
#   → ノードのリソースが不足。kubectl describe nodes でリソース確認
kubectl describe nodes | grep -A 5 "Allocated resources"

# "no nodes available to schedule pods"
#   → taint/toleration の不一致
kubectl describe node <node> | grep Taints

# "pod has unbound immediate PersistentVolumeClaims"
#   → PVC が Bound になっていない
kubectl get pvc
```

## Service にアクセスできない

Pod は動いているのに HTTP リクエストが届かない。

```bash
# 1. Service のセレクターと Pod のラベルが一致しているか確認
kubectl get svc <service> -o yaml | grep selector
kubectl get pods --show-labels

# 2. Endpoints が存在するか確認（ここが空なら Pod に到達していない）
kubectl get endpoints <service>

# 3. Pod の中から Service に疎通できるか確認
kubectl exec -it <pod> -- curl http://<service-name>:<port>

# 4. DNS 解決できるか
kubectl exec -it <pod> -- nslookup <service-name>
```

## ImagePullBackOff

コンテナイメージが取得できない。

```bash
kubectl describe pod <pod>
# Events に "Failed to pull image" の詳細

# よくある原因:
# 1. イメージ名・タグのタイポ
# 2. プライベートレジストリの認証情報がない
#    → imagePullSecrets の設定を確認
kubectl get pod <pod> -o jsonpath='{.spec.imagePullSecrets}'

# imagePullSecrets を作成
kubectl create secret docker-registry regcred \
  --docker-server=myregistry.example.com \
  --docker-username=myuser \
  --docker-password=mypassword
```

## Deployment のロールアウト状況を確認する

```bash
# ロールアウトの状態確認
kubectl rollout status deployment/myapp

# ロールアウト履歴
kubectl rollout history deployment/myapp

# 直前のバージョンにロールバック
kubectl rollout undo deployment/myapp

# 特定のリビジョンにロールバック
kubectl rollout undo deployment/myapp --to-revision=3
```

---

## リソース使用状況を確認する

```bash
# ノードのリソース使用率
kubectl top nodes

# Pod のリソース使用率
kubectl top pods --all-namespaces

# 特定 namespace の Pod
kubectl top pods -n production
```

## 便利なデバッグ用コマンド集

```bash
# 全 namespace の異常な Pod を一覧
kubectl get pods -A | grep -v Running | grep -v Completed

# イベントを時系列で確認（直近の障害原因を探す）
kubectl get events --sort-by='.lastTimestamp' -n production

# Pod のシェルに入る
kubectl exec -it <pod> -- /bin/sh

# 一時的なデバッグコンテナを起動（Pod を変更せずにデバッグ）
kubectl debug -it <pod> --image=busybox --target=<container>

# ポートフォワードでローカルから直接アクセス
kubectl port-forward pod/<pod> 8080:8080

# ConfigMap・Secret の内容を確認
kubectl get configmap <name> -o yaml
kubectl get secret <name> -o jsonpath='{.data.password}' | base64 -d
```
