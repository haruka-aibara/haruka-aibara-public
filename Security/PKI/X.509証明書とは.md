# X.509 証明書とは

## どこで出てくるか

- HTTPS でブラウザが南京錠マークを表示するとき
- AWS IoT Core でデバイスを認証するとき
- OTA アップデートのファームウェアに署名するとき
- VPN やゼロトラストでクライアント証明書を使うとき

全部 X.509 証明書が使われている。仕組みを整理する。

---

## 解決したい問題

インターネットで通信するとき、相手が名乗っている通りの存在か確認する方法がない。

- `example.com` に接続したとき、本当にそのサーバーか？
- IoT デバイスが「自分は VIN xxxxx の車両です」と言ってきたとき、本当にその車か？

パスワードは「知っている情報」で証明する。でもパスワードは盗まれる・漏れる。X.509 証明書は「持っている鍵のペア」で証明する。

---

## 仕組みの核心：公開鍵暗号

X.509 の土台は**公開鍵暗号**（非対称暗号）だ。

```
秘密鍵（private key）：自分だけが持つ。絶対に渡さない
公開鍵（public key） ：誰に渡してもいい
```

この 2 つには特殊な関係がある。

- 秘密鍵で署名したものは、対応する公開鍵でしか検証できない
- 公開鍵で暗号化したものは、対応する秘密鍵でしか復号できない

「秘密鍵を持っている = 本人である」という証明が数学的に成立する。

---

## X.509 証明書は「公開鍵の身分証明書」

公開鍵だけ渡されても「これが誰の鍵かわからない」問題がある。X.509 証明書は公開鍵に**名前・有効期限・発行者**などの情報を加えて、第三者（CA）が署名したもの。

証明書の中身（主要フィールド）：

```
Subject（誰の証明書か）  : CN=honda.com
Issuer（誰が発行したか） : CN=Amazon Root CA 1
Public Key              : (公開鍵のデータ)
Valid From              : 2026-01-01
Valid To                : 2027-01-01
Serial Number           : 0A:1B:2C:...
Signature               : (CA の秘密鍵による署名)
```

---

## CA（認証局）と信頼の連鎖

X.509 の核心は「CA を信頼する」という仕組みにある。

```
Root CA（ルート認証局）
  └── Intermediate CA（中間認証局）
        └── サーバー証明書 / デバイス証明書
```

ブラウザや OS は「信頼できる Root CA のリスト」を持っている。Root CA が署名した Intermediate CA の証明書、その Intermediate CA が署名したサーバー証明書、という連鎖（チェーン）を辿ることで最終的に信頼を検証する。

この連鎖を **証明書チェーン（Certificate Chain）** と呼ぶ。

---

## TLS ハンドシェイクで何が起きているか

HTTPS 通信の最初に行われる TLS ハンドシェイクを X.509 の観点で追う。

```
1. クライアント → サーバー : 「繋ぎたい、こういう暗号が使えます」
2. サーバー → クライアント : 「わかった、あとサーバー証明書を渡すね」
3. クライアント             : 証明書チェーンを検証する
                             ・有効期限は切れていないか
                             ・信頼できる CA が署名しているか
                             ・接続先のドメイン名と一致するか
4. クライアント → サーバー : 証明書内の公開鍵で暗号化して共通鍵を送る
5. 以降                    : 共通鍵で暗号化した通信
```

ステップ 4 が重要で、「公開鍵で暗号化 → 秘密鍵でしか復号できない」という性質を使っている。秘密鍵を持っているサーバーだけが復号できるため、「本物のサーバーである」ことが証明される。

---

## クライアント証明書：逆方向の認証

通常の TLS は「サーバーが本物か」をクライアントが検証する。クライアント証明書はその逆で、「クライアント（デバイス）が本物か」をサーバーが検証する。

```
通常の TLS           : サーバーが証明書を提示 → クライアントが検証
相互 TLS（mTLS）     : サーバーとクライアントの両方が証明書を提示・検証
```

IoT デバイス認証は mTLS が基本だ。パスワードやトークンを使わず、デバイスに焼き込んだ秘密鍵・証明書ペアで認証する。

## AWS での X.509 の登場場面

| サービス | 用途 |
|---|---|
| ACM（Certificate Manager） | ALB・CloudFront 向けのサーバー証明書を無料で発行・自動更新 |
| ACM Private CA | 社内用・IoT 用のプライベート PKI を構築。独自の Root CA を持てる |
| AWS IoT Core | デバイスを X.509 証明書で認証。Thing ごとに証明書を発行 |
| AWS Signer | Lambda・IoT デバイスへのコードに署名。改ざん検知に使う |

### ACM と ACM Private CA の違い

**ACM（パブリック証明書）**：パブリック CA が署名。ブラウザがそのまま信頼する。HTTPS サイト向け。

**ACM Private CA**：自分が Root CA になる。外部からは信頼されないが、自社デバイスや社内システムには使える。IoT フリートや VPN クライアント証明書に向いている。

---

## SDV での使われ方

車両 1 台 = 証明書 1 枚という構造になる。

```
[工場]
  │ 車両製造時に ACM Private CA から証明書を発行
  │ ECU の HSM（ハードウェアセキュリティモジュール）に秘密鍵を格納
  │ 秘密鍵は HSM 外に出ない
  ↓
[車両運用中]
  車両 → (mTLS) → AWS IoT Core
  証明書の Subject = VIN（車両識別番号）
  ↓
[OTA アップデート]
  OEM のサーバーが秘密鍵でファームウェアに署名
  車両側が公開鍵で署名を検証 → 正規の OEM が作ったファームウェアか確認
```

10〜15 年の車両寿命を考えると、証明書の有効期限管理とアルゴリズムの更新（SHA-256 が将来も安全か）を設計段階から考える必要がある。

---

## 証明書の失効

証明書が漏洩・不正使用されたとき、有効期限を待たずに無効化する仕組みが 2 つある。

- **CRL（Certificate Revocation List）**：失効した証明書のリストを CA が定期公開。古い方式
- **OCSP（Online Certificate Status Protocol）**：証明書の有効性をリアルタイムに問い合わせる

IoT フリートでは数百万台分の証明書管理が必要になるため、失効フローの設計は PKI 設計の重要なポイントになる。
