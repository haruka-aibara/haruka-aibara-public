# 動的シークレット

「DB パスワードをローテーションしたいが、どのサービスが使っているか把握できていない」「本番 DB のパスワードが社内の全エンジニアに知られている」——Vault の動的シークレットは、アプリが必要なときだけ一時的な認証情報を発行し、TTL が切れると自動で無効化する。

---

## 動的シークレットの仕組み

```
アプリ → Vault に「DB の認証情報をくれ」
Vault → PostgreSQL に「専用ユーザーを作れ」（TTL: 1時間）
Vault → アプリに一時ユーザー名とパスワードを返す
1時間後 → Vault が PostgreSQL から専用ユーザーを自動削除
```

- 認証情報が漏洩しても TTL が切れれば無効
- 誰がいつ使ったかを Vault が記録（監査ログ）
- 共有パスワードが存在しない

---

## PostgreSQL 動的シークレットの設定

```bash
# Database シークレットエンジンを有効化
vault secrets enable database

# PostgreSQL 接続設定
vault write database/config/mydb \
  plugin_name=postgresql-database-plugin \
  connection_url="postgresql://{{username}}:{{password}}@db:5432/mydb" \
  allowed_roles="myapp-role" \
  username="vault_admin" \
  password="vault_admin_pass"

# ロールの定義（TTL と SQL を指定）
vault write database/roles/myapp-role \
  db_name=mydb \
  creation_statements="CREATE ROLE \"{{name}}\" WITH LOGIN PASSWORD '{{password}}' VALID UNTIL '{{expiration}}';
    GRANT SELECT, INSERT, UPDATE ON ALL TABLES IN SCHEMA public TO \"{{name}}\";" \
  default_ttl="1h" \
  max_ttl="24h"
```

## アプリから認証情報を取得する

```bash
# CLI で取得
vault read database/creds/myapp-role

# Key: Value
# lease_id:       database/creds/myapp-role/xxxx
# lease_duration: 1h
# username:       v-myapp-xxxx
# password:       A1b2C3d4-xxxx
```

```python
import hvac

client = hvac.Client(url='http://vault:8200', token='s.xxxx')
creds = client.secrets.database.generate_credentials(name='myapp-role')

db_user = creds['data']['username']
db_pass = creds['data']['password']
# → この認証情報で DB に接続する
```

## AWS 動的シークレット（IAM ユーザーを一時的に発行）

```bash
# AWS シークレットエンジンを有効化
vault secrets enable aws

# Vault に AWS 管理用の認証情報を設定
vault write aws/config/root \
  access_key=AKIAIOSFODNN7EXAMPLE \
  secret_key=wJalrXUtnFEMI/K7MDENG/bPxRfiCYEXAMPLEKEY \
  region=ap-northeast-1

# ロール設定（S3 読み取りのみ）
vault write aws/roles/s3-reader \
  credential_type=iam_user \
  policy_document=-<<EOF
{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Action": ["s3:GetObject"],
    "Resource": "arn:aws:s3:::my-bucket/*"
  }]
}
EOF

# 一時 IAM ユーザーを発行
vault read aws/creds/s3-reader
```

## Kubernetes からの認証（Kubernetes Auth Method）

Kubernetes の Pod が Vault に認証する。サービスアカウントトークンを使うため、長期的なシークレットが不要。

```bash
# Kubernetes Auth を有効化
vault auth enable kubernetes

# Kubernetes クラスターの情報を設定
vault write auth/kubernetes/config \
  kubernetes_host="https://kubernetes.default.svc:443" \
  kubernetes_ca_cert=@/var/run/secrets/kubernetes.io/serviceaccount/ca.crt

# ロール設定
vault write auth/kubernetes/role/myapp \
  bound_service_account_names=myapp \
  bound_service_account_namespaces=production \
  policies=myapp-policy \
  ttl=1h
```

```yaml
# Pod の Vault Agent サイドカーでシークレットを自動取得
annotations:
  vault.hashicorp.com/agent-inject: "true"
  vault.hashicorp.com/role: "myapp"
  vault.hashicorp.com/agent-inject-secret-config: "database/creds/myapp-role"
```

## AWS Secrets Manager との使い分け

| | Vault | AWS Secrets Manager |
|---|---|---|
| 動的シークレット | ◎ 多数の DB・クラウドに対応 | △ RDS のみ（自動ローテーション） |
| マルチクラウド | ◎ AWS・GCP・Azure 対応 | × AWS のみ |
| 自己ホスト | ◎ VPC 内に閉じられる | × AWS 管理 |
| 運用コスト | △ Vault クラスタの管理が必要 | ◎ フルマネージド |
| コスト | ◎（OSS は無料） | △ シークレット単位の月額課金 |

マルチクラウド・マルチ DB 環境なら Vault、AWS だけなら Secrets Manager が楽。
