# 最小権限の原則

「とりあえず AdministratorAccess を付けた」「よくわからないから広めに付けた」——こうしてつけた過剰な権限が、アカウント侵害時の被害を最大化する。最小権限の原則は、必要最低限の権限だけを付与することで、侵害された場合の影響範囲を限定する考え方。

---

## なぜ最小権限が重要か

EC2 インスタンスに付けた IAM ロールが `AdministratorAccess` だった場合、そのインスタンスが侵害されると攻撃者はアカウント全体を掌握できる。最小権限なら、そのインスタンスが使う S3 バケットと DynamoDB テーブルへのアクセスだけに被害が限定される。

---

## 実践的なアプローチ

### 1. AWS 管理ポリシーから始めてカスタマイズする

最初から完璧なカスタムポリシーを書こうとしない。まず `AmazonS3ReadOnlyAccess` などの AWS 管理ポリシーを付け、不要な権限を削る方向でカスタマイズする。

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:ListBucket"
      ],
      "Resource": [
        "arn:aws:s3:::my-app-bucket",
        "arn:aws:s3:::my-app-bucket/*"
      ]
    }
  ]
}
```

### 2. IAM Access Analyzer でポリシーを生成する

実際に使ったアクセスログ（CloudTrail）から、必要な権限だけのポリシーを自動生成できる。

1. CloudTrail を有効にして一定期間使う
2. IAM コンソール → Access Analyzer → ポリシー生成
3. エンティティを選択して「ポリシーを生成」

実運用で使ったアクセスだけが含まれるので、最小権限に近いポリシーが作れる。

### 3. リソースレベルで絞る

アクション（何ができるか）だけでなく、対象リソース（どのリソースに）も絞る。

```json
{
  "Effect": "Allow",
  "Action": "s3:GetObject",
  "Resource": "arn:aws:s3:::my-specific-bucket/*"
  // "Resource": "*" は NG
}
```

### 4. 条件（Condition）で制限を加える

MFA 認証済みの場合だけ許可、特定の IP アドレスからのみ許可、といった条件を付けられる。

```json
{
  "Effect": "Allow",
  "Action": "iam:DeleteUser",
  "Resource": "*",
  "Condition": {
    "Bool": {
      "aws:MultiFactorAuthPresent": "true"
    }
  }
}
```

## 権限の定期的な見直し

付けた権限は定期的に棚卸しする。IAM コンソールの「最終使用日時」を確認して、長期間使われていない権限を削る。

```bash
# IAM ユーザーの最終使用日時を確認
aws iam get-user --user-name alice
aws iam list-access-keys --user-name alice

# ロールの最終使用日時
aws iam get-role --role-name my-role
```

IAM Access Analyzer の「未使用のアクセス」分析を使うと、使われていない権限・ロール・アクセスキーをまとめて検出できる。
