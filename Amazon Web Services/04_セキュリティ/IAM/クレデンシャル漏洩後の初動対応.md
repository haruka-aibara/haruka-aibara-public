# クレデンシャル漏洩後の初動対応

GitHub に AWS アクセスキーを push してしまった。.env ファイルを誤って公開した。退職者のキーを無効化し忘れた——クレデンシャル漏洩は AWS セキュリティインシデントの中で最も頻発するケースだ。

問題は「漏洩した」という事実より、**漏洩後の対応が遅れることで被害が拡大する**こと。GitHub にキーを push してから悪用されるまで数秒〜数分という事例が報告されている。スキャンボットが常時 GitHub を監視しているためだ。

対応手順を事前に知っておくかどうかで、被害の規模が決まる。

---

## 最初の15分でやること

### 0分目：キーを即座に無効化（削除ではなく無効化）

```bash
# 無効化（証拠を残すため削除しない）
aws iam update-access-key \
  --access-key-id AKIA... \
  --status Inactive \
  --user-name target-user
```

削除してしまうと CloudTrail のログと突合できなくなる。必ず無効化（Inactive）にとどめる。

### 2分目：アクティブなセッションをすべて失効させる

アクセスキーを無効化しても、そのキーで発行済みの STS 一時トークンは有効期限まで生き続ける。ポリシーで即時ブロックする。

```bash
# 全アクションを即時 Deny（既存セッションも無効化される）
aws iam put-user-policy \
  --user-name target-user \
  --policy-name EmergencyDenyAll \
  --policy-document '{
    "Version": "2012-10-17",
    "Statement": [{
      "Effect": "Deny",
      "Action": "*",
      "Resource": "*"
    }]
  }'
```

IAM ロールのセッションを失効させる場合：

```bash
# ロールのセッションを無効化（AWSRevokeOlderSessions を使う）
aws iam put-role-policy \
  --role-name target-role \
  --policy-name RevokeOlderSessions \
  --policy-document '{
    "Version": "2012-10-17",
    "Statement": [{
      "Effect": "Deny",
      "Action": "*",
      "Resource": "*",
      "Condition": {
        "DateLessThan": {
          "aws:TokenIssueTime": "'$(date -u +%Y-%m-%dT%H:%M:%SZ)'"
        }
      }
    }]
  }'
```

### 5分目：何をやられたか調査開始

CloudTrail で当該キーの全操作履歴を引く。

```bash
# 過去7日間の当該アクセスキーの操作を確認
aws cloudtrail lookup-events \
  --lookup-attributes AttributeKey=AccessKeyId,AttributeValue=AKIA... \
  --start-time $(date -u -d '7 days ago' +%Y-%m-%dT%H:%M:%SZ) \
  --query 'Events[*].[EventTime,EventName,SourceIPAddress,Username]' \
  --output table
```

確認すること：
- いつから悪用されていたか（漏洩タイミングの特定）
- どのリージョンからアクセスがあったか（普段と違う地域は要注意）
- 何の操作をされたか（IAM 操作・EC2 起動・S3 アクセス等）

### 10分目：被害の横展開を確認

攻撃者が最初にやることは「権限昇格」と「永続化」。以下を確認する。

```bash
# 新しく作られた IAM ユーザー・ロールがないか
aws iam list-users --query 'Users[?CreateDate>=`2024-01-01`]'

# 新しくアタッチされたポリシーがないか（過去24時間）
aws cloudtrail lookup-events \
  --lookup-attributes AttributeKey=EventName,AttributeValue=AttachUserPolicy \
  --start-time $(date -u -d '1 day ago' +%Y-%m-%dT%H:%M:%SZ)

# 見知らぬリージョンで EC2 が起動されていないか（クリプトマイニング）
for region in $(aws ec2 describe-regions --query 'Regions[].RegionName' --output text); do
  count=$(aws ec2 describe-instances --region $region \
    --filters Name=instance-state-name,Values=running \
    --query 'length(Reservations[].Instances[])' --output text)
  if [ "$count" -gt "0" ]; then
    echo "$region: $count instances"
  fi
done
```

### 15分目：被害範囲を確定して報告

以下を整理して関係者に共有する：
- 漏洩したキーの所有者と権限範囲
- 漏洩が疑われる時刻
- 確認された不正操作の一覧
- すでに取った対処（無効化・セッション失効）
- 次のアクション（新キー発行・再発防止策）

## GitHub に push してしまった場合の追加対応

GitHub のシークレットスキャン機能が検知する前に攻撃される可能性がある。

```bash
# git の履歴からキーを削除（BFG Repo Cleaner または git filter-repo）
git filter-repo --replace-text <(echo 'AKIA...**REMOVED**')

# または BFG を使う
java -jar bfg.jar --replace-text passwords.txt my-repo.git
```

重要：**git 履歴から削除しても、それまでの時間に悪用されている可能性は消えない**。キーの無効化を先にやってから履歴を整理する。

---

## 根本対策：キーを使わない構成にする

長期クレデンシャル（アクセスキー）は漏洩リスクが常にある。以下の構成に移行することが最終的な解決策。

| 用途 | キーレス認証の方法 |
|---|---|
| GitHub Actions → AWS | OIDC（aws-actions/configure-aws-credentials） |
| EC2 上のアプリ → AWS | IAM インスタンスプロファイル |
| Lambda → AWS | 実行ロール |
| ローカル開発 → AWS | IAM Identity Center（SSO）の一時トークン |

一度移行してしまえば「キーが漏洩する」というリスクそのものがなくなる。
