# ログ監視ロードマップ

「何から手をつければいいかわからない」——AWS のログ管理・脅威検知は選択肢が多すぎて、全部いっぺんにやろうとすると何もできないまま終わる。

AWS セキュリティ成熟度モデルの4フェーズに沿って、優先順位をつけて進める道筋を整理する。

| フェーズ | 目安期間 | 目標 |
|---|---|---|
| **Quick Wins** | 1〜2週間 | 少ない工数で即効果。まずここから |
| **Foundational** | 1〜3ヶ月 | 最低限必要なログ基盤を整える |
| **Efficient** | 3〜6ヶ月 | 運用を回せる状態にする |
| **Optimized** | 継続的 | 自動化と継続改善 |

---

## Quick Wins：1〜2週間でできること

工数がほぼかからず、有効化するだけで大きな効果が得られるもの。後回しにする理由がない。

### GuardDuty を有効化する

ML ベースの脅威検知が設定ゼロで動き出す。CloudTrail・VPC Flow Logs・DNS クエリログを自動で分析して、C2 通信・クレデンシャル漏洩・異常な API 呼び出し等を検知してくれる。

```bash
aws guardduty create-detector --enable --region ap-northeast-1
```

有効化するだけでよい。ルールを書く必要はない。

### Security Hub を有効化する

GuardDuty・Inspector・Macie・Config・IAM Access Analyzer の findings が一箇所に集まる。AWS Foundational Security Best Practices（FSBP）の自動チェックも動き出す。

```bash
aws securityhub enable-security-hub \
  --enable-default-standards \
  --region ap-northeast-1
```

### Slack にアラートを飛ばす

Security Hub の findings を EventBridge → AWS Chatbot → Slack に流す。Lambda 不要で繋がる。

```bash
# Chatbot の設定はコンソールから行う（CLIより分かりやすい）
# Security Hub → EventBridge ルール → Chatbot → Slack チャンネル
```

CRITICAL・HIGH 以上の findings だけ通知するように EventBridge のルールで絞るのがポイント。全部流すとすぐにアラート疲れになる。

### CloudTrail が有効になっているか確認する

ほとんどのアカウントで有効だが、全リージョン・マルチリージョン証跡になっているか確認する。

```bash
aws cloudtrail describe-trails --include-shadow-trails false
```

S3 への保存先が決まっていない場合はこのタイミングで設定する。

## Foundational：1〜3ヶ月でやること

ログを「集める」基盤を整える。この段階が整っていないと後のフェーズが成立しない。

### Security Lake を有効化する（ログのホーム）

全 AWS ログを OCSF 形式で S3 に正規化して保存する基盤。後で SIEM を変えても生ログはここに残るので、**早く有効化するほど溜まるログが増える**。

有効化で自動収集されるもの：CloudTrail・VPC Flow Logs・Route53 クエリログ・Security Hub findings・Lambda 実行ログ（オプション）

### VPC Flow Logs を全 VPC で有効化する

デフォルトで無効。有効化しないと「どの IP がどこに通信したか」が永久に追えない。

```bash
aws ec2 create-flow-logs \
  --resource-type VPC \
  --resource-ids vpc-xxxxxx \
  --traffic-type ALL \
  --log-destination-type s3 \
  --log-destination arn:aws:s3:::your-flow-log-bucket
```

Security Lake を有効化していれば自動で収集される。

### GuardDuty のオプション保護を有効化する

デフォルトの GuardDuty は EC2・CloudTrail・DNS だけ。追加オプションで対象が広がる。

```bash
# S3 保護（S3 への不審なアクセスを検知）
aws guardduty update-detector \
  --detector-id <id> \
  --features '[{"Name":"S3_DATA_EVENTS","Status":"ENABLED"}]'

# EKS 保護・Lambda 保護・RDS 保護も同様に有効化
```

### アラート対応プレイブックを最低限書く

Slack にアラートが来たとき「で、どうするの？」が決まっていないと意味がない。

GuardDuty の finding タイプは 200 種類以上あるが、対応パターンは実質 6 つに収束する。finding タイプごとの詳細な推奨対応は [AWS 公式ドキュメント](https://docs.aws.amazon.com/guardduty/latest/ug/guardduty_finding-types-active.html) に全種類分掲載されているので、各プレイブックからリンクで参照する形にするのが現実的。

---

#### パターン① EC2 が侵害された疑い

**該当 finding の例**：`Backdoor:EC2/C&CActivity`・`CryptoCurrency:EC2/BitcoinTool`・`Trojan:EC2/*`・`UnauthorizedAccess:EC2/MaliciousIPCaller` 等

```
1. 対象インスタンスの特定
   → finding の detail から instanceId を確認

2. インスタンスを隔離（通信を遮断）
   → セキュリティグループを「アウトバウンドすべて拒否」のものに差し替え
   → Auto Scaling グループから切り離す（新しいインスタンスに置き換えさせる）

3. フォレンジック用スナップショットを取得
   → EBS ボリュームをスナップショット（証拠保全）

4. 調査
   → CloudTrail で当該インスタンスからの API 呼び出しを確認
   → VPC Flow Logs で外部への不審な通信先を確認
   → SSM Session Manager でプロセス・ファイルを確認（SSH 不要）

5. 終息
   → 感染インスタンスを terminate
   → AMI・ユーザーデータ・起動設定に不審なものがないか確認してから新インスタンスを起動
```

参考：[Remediating a potentially compromised Amazon EC2 instance](https://docs.aws.amazon.com/guardduty/latest/ug/guardduty_remediate.html#compromised-ec2)

---

#### パターン② IAM クレデンシャル漏洩

**該当 finding の例**：`UnauthorizedAccess:IAMUser/MaliciousIPCaller`・`CredentialAccess:IAMUser/AnomalousBehavior`・`Recon:IAMUser/*`・`Persistence:IAMUser/*` 等

```
1. 漏洩したキー・セッションを特定
   → finding の detail から accessKeyId・userName を確認

2. 即座に無効化
   → IAM アクセスキーを無効化（削除ではなく無効化で証拠を残す）
   aws iam update-access-key \
     --access-key-id AKIA... \
     --status Inactive \
     --user-name target-user

3. アクティブなセッションを失効
   → IAM ロールの場合：セッションポリシーで既存セッションを無効化
   aws iam put-user-policy --user-name target-user \
     --policy-name DenyAll \
     --policy-document '{"Version":"2012-10-17","Statement":[{"Effect":"Deny","Action":"*","Resource":"*"}]}'

4. 被害範囲の調査
   → CloudTrail で当該キーの全 API 呼び出しを確認（いつから・何を・どこから）
   → 新たに作成された IAM ユーザー・ロール・ポリシーがないか確認
   → S3・RDS・EC2 等への不審なアクセスがないか確認

5. 終息・再発防止
   → 新しいキーを発行（必要な場合）
   → キーの保管・取り扱い方法を見直す
   → OIDC 等でキーレス認証に移行できないか検討
```

参考：[Remediating compromised AWS credentials](https://docs.aws.amazon.com/guardduty/latest/ug/guardduty_remediate.html#compromised-creds)

---

#### パターン③ S3 への不審アクセス

**該当 finding の例**：`Discovery:S3/MaliciousIPCaller`・`Exfiltration:S3/MaliciousIPCaller`・`Impact:S3/MaliciousIPCaller`・`UnauthorizedAccess:S3/*` 等

```
1. 対象バケット・オブジェクトの特定
   → finding の detail から bucketName・object.key を確認

2. アクセス元の確認
   → CloudTrail（S3 データイベント）で当該バケットへの全アクセスを確認
   → 不審な IP・IAM エンティティを特定

3. 被害確認
   → 機密データが含まれているバケットか確認（Macie でスキャン）
   → オブジェクトが削除・変更・外部コピーされていないか確認

4. 遮断
   → 不審なアクセス元 IP をバケットポリシーで明示的に Deny
   → パブリックアクセスが意図せず開いていた場合は即時ブロック
   aws s3api put-public-access-block \
     --bucket target-bucket \
     --public-access-block-configuration "BlockPublicAcls=true,IgnorePublicAcls=true,BlockPublicPolicy=true,RestrictPublicBuckets=true"

5. 調査・通知
   → 個人情報・機密情報が流出した可能性があれば法務・コンプライアンスに報告
```

---

#### パターン④ ポートスキャン・偵察

**該当 finding の例**：`Recon:EC2/PortProbeUnprotectedPort`・`Recon:EC2/Portscan`・`Discovery:*` 等

```
1. 意図的かどうかを判断
   → ペネトレーションテスト中か？ → サプレッションルールを作成
   → バスティオンホスト等の意図的に公開しているポートか？ → 同上

2. 意図的でない場合
   → セキュリティグループを見直して不要なポートを閉じる
   → 22/3389 は会社の IP レンジのみに制限する

3. サプレッションルールの作成（意図的な場合）
   → Finding type + Instance ID/Tag の条件で抑制
```

---

#### パターン⑤ マイニング・C2 通信（高確度の侵害）

**該当 finding の例**：`CryptoCurrency:*`・`Backdoor:EC2/C&CActivity`・`Trojan:EC2/BlackholeTraffic` 等

severity が HIGH で出る場合はほぼ確実に侵害。**パターン①（EC2 侵害）の手順に従い、調査より先に隔離を優先する。**

---

#### パターン⑥ 業務上の正常動作（誤検知）

**該当 finding の例**：ペネトレーションテストツール・セキュリティスキャナー・社内からの意図的なアクセス等

```
1. 業務上の正常動作であることを確認・記録

2. GuardDuty のサプレッションルールを作成
   → Finding type + 対象リソースの条件で絞る（広すぎる抑制は危険）
   aws guardduty create-filter \
     --detector-id <id> \
     --name "suppress-pentest-ec2" \
     --action ARCHIVE \
     --finding-criteria '{"Criterion": {"type": {"Eq": ["Recon:EC2/PortProbeUnprotectedPort"]}, "resource.instanceDetails.tags.key": {"Eq": ["Environment"]}, "resource.instanceDetails.tags.value": {"Eq": ["pentest"]}}}'

3. 抑制した内容を記録しておく（後で見たときに「なぜ抑制したか」が分かるように）
```

#### プレイブック共通事項

**誰が対応するか・エスカレーション先を決めておく：**

```
一次対応（アラート確認・初動）：インフラ担当
エスカレーション（侵害が疑われる場合）：セキュリティ担当 or マネージャー
法的判断が必要（データ漏洩の可能性）：法務・コンプライアンス
```

finding タイプ別の詳細な AWS 推奨対応：
- [EC2 finding types](https://docs.aws.amazon.com/guardduty/latest/ug/guardduty_finding-types-ec2.html)
- [IAM finding types](https://docs.aws.amazon.com/guardduty/latest/ug/guardduty_finding-types-iam.html)
- [S3 finding types](https://docs.aws.amazon.com/guardduty/latest/ug/guardduty_finding-types-s3.html)
- [EKS finding types](https://docs.aws.amazon.com/guardduty/latest/ug/guardduty_finding-types-eks.html)
- [Runtime finding types](https://docs.aws.amazon.com/guardduty/latest/ug/guardduty_finding-types-runtime-monitoring.html)

## Efficient：3〜6ヶ月でやること

「集めて通知する」から「調査できる・運用が回る」状態にする。

### Athena で Security Lake のログを分析できるようにする

Security Lake に溜まったログを SQL でクエリできる。「この IP が過去1ヶ月でどう動いたか」を調べられるようになる。

```sql
-- 例：特定 IP からの CloudTrail API 呼び出しを確認
SELECT eventtime, eventsource, eventname, sourceipaddress
FROM amazon_security_lake_glue_db_ap_northeast_1.amazon_security_lake_table_ap_northeast_1_cloud_trail_mgmt_1_0
WHERE sourceipaddress = '1.2.3.4'
  AND eventtime > now() - interval '7' day
ORDER BY eventtime DESC
```

インシデント調査の基本ツールになる。

### アラートのチューニング（false positive を削減する）

Security Hub の findings は最初から大量に出る。対応できないほど通知が多いとアラート疲れになり、本物の脅威を見落とす。

```bash
# 対応済み・意図的な設定は Suppressed にする
aws securityhub batch-update-findings \
  --finding-identifiers '[{"ProductArn":"...","Id":"..."}]' \
  --workflow '{"Status":"SUPPRESSED"}' \
  --note '{"Text":"意図的な設定のため抑制","UpdatedBy":"your-name"}'
```

月次でレビューして「常に出てるが問題ないもの」を仕分けていく作業。これを続けることで「鳴ったら本物」な状態に近づく。

### Inspector で ECR・EC2 を継続スキャンする

GuardDuty が「不審な挙動」を見るのに対して、Inspector は「既知の脆弱性（CVE）」を見る。

```bash
aws inspector2 enable \
  --resource-types EC2 ECR LAMBDA \
  --region ap-northeast-1
```

ECR にイメージを push するたびに自動スキャンが走り、CRITICAL な CVE があれば Security Hub に findings が上がる。

### 対応の一部を自動化する

アラートごとに人間が手で同じ操作をするのは続かない。繰り返し発生する対応は Lambda で自動化する。

```
例：
- 不審な IAM アクセスキーの検知 → 自動でキーを無効化
- 外部公開 S3 バケットの検知 → 自動でブロックパブリックアクセスをオン
- 感染が疑われる EC2 → 自動でセキュリティグループを隔離用に変更
```

全部自動化するのではなく「明らかに問題で、誤検知リスクが低い」ものから始める。

## Optimized：継続的にやること

この段階は専任のセキュリティエンジニアがいることが前提。いない場合は Efficient フェーズを磨き続けるほうが現実的。

### 検知ルールを Detection as Code で管理する

検知ルール（EventBridge ルール・OpenSearch クエリ・Lambda ロジック）を Git で管理して CI/CD でデプロイする。

- ルールの変更が追跡できる
- レビュープロセスを挟める
- 環境間でルールを一致させられる

### スレットハンティングを定期的に行う

アラートが来るのを待つのではなく、能動的に「怪しい動きがないか」を探す。月次や四半期で実施する。

Security Lake + Athena、または SIEM のクエリ画面で「過去30日間で普段と違う動きをした IAM ロールはあるか」「深夜に API を叩いているユーザーはいるか」のような仮説を立てて調べる。

### SIEM の導入を検討する（必要であれば）

Efficient フェーズを6ヶ月以上回して「GuardDuty の検知では足りない」「カスタムの相関ルールが必要」「スレットハンティングの効率を上げたい」という具体的な課題が見えてきたタイミングで検討する。

選択肢と判断軸は [ログ集約と SIEM 選定](ログ集約とSIEM選定.md) を参照。

## フェーズ別チェックリスト

### Quick Wins
- [ ] GuardDuty 有効化（全リージョン）
- [ ] Security Hub 有効化（FSBP 有効）
- [ ] CloudTrail 全リージョン証跡を確認
- [ ] Slack アラート通知設定（CRITICAL/HIGH のみ）

### Foundational
- [ ] Security Lake 有効化
- [ ] VPC Flow Logs 全 VPC で有効化
- [ ] GuardDuty オプション保護（S3・EKS・Lambda・RDS）
- [ ] アラート対応プレイブック（最低限）作成

### Efficient
- [ ] Athena で Security Lake クエリができる状態
- [ ] false positive の定期レビュー・抑制運用
- [ ] Inspector 有効化（ECR・EC2・Lambda）
- [ ] 高頻度対応の自動化（最低1〜2件）

### Optimized
- [ ] Detection as Code（GitOps 管理）
- [ ] 定期スレットハンティング（月次以上）
- [ ] SIEM 導入判断と運用
