# ログ監視ロードマップ

「何から手をつければいいかわからない」——AWS のログ管理・脅威検知は選択肢が多すぎて、全部いっぺんにやろうとすると何もできないまま終わる。

AWS セキュリティ成熟度モデルの4フェーズに沿って、優先順位をつけて進める道筋を整理する。

| フェーズ | 目安期間 | 目標 |
|---|---|---|
| **Quick Wins** | 1〜2週間 | 少ない工数で即効果。まずここから |
| **Foundational** | 1〜3ヶ月 | 最低限必要なログ基盤を整える |
| **Efficient** | 3〜6ヶ月 | 運用を回せる状態にする |
| **Optimized** | 継続的 | 自動化と継続改善 |

---

## Quick Wins：1〜2週間でできること

工数がほぼかからず、有効化するだけで大きな効果が得られるもの。後回しにする理由がない。

### GuardDuty を有効化する

ML ベースの脅威検知が設定ゼロで動き出す。CloudTrail・VPC Flow Logs・DNS クエリログを自動で分析して、C2 通信・クレデンシャル漏洩・異常な API 呼び出し等を検知してくれる。

```bash
aws guardduty create-detector --enable --region ap-northeast-1
```

有効化するだけでよい。ルールを書く必要はない。

### Security Hub を有効化する

GuardDuty・Inspector・Macie・Config・IAM Access Analyzer の findings が一箇所に集まる。AWS Foundational Security Best Practices（FSBP）の自動チェックも動き出す。

```bash
aws securityhub enable-security-hub \
  --enable-default-standards \
  --region ap-northeast-1
```

### Slack にアラートを飛ばす

Security Hub の findings を EventBridge → AWS Chatbot → Slack に流す。Lambda 不要で繋がる。

```bash
# Chatbot の設定はコンソールから行う（CLIより分かりやすい）
# Security Hub → EventBridge ルール → Chatbot → Slack チャンネル
```

CRITICAL・HIGH 以上の findings だけ通知するように EventBridge のルールで絞るのがポイント。全部流すとすぐにアラート疲れになる。

### CloudTrail が有効になっているか確認する

ほとんどのアカウントで有効だが、全リージョン・マルチリージョン証跡になっているか確認する。

```bash
aws cloudtrail describe-trails --include-shadow-trails false
```

S3 への保存先が決まっていない場合はこのタイミングで設定する。

## Foundational：1〜3ヶ月でやること

ログを「集める」基盤を整える。この段階が整っていないと後のフェーズが成立しない。

### Security Lake を有効化する（ログのホーム）

全 AWS ログを OCSF 形式で S3 に正規化して保存する基盤。後で SIEM を変えても生ログはここに残るので、**早く有効化するほど溜まるログが増える**。

有効化で自動収集されるもの：CloudTrail・VPC Flow Logs・Route53 クエリログ・Security Hub findings・Lambda 実行ログ（オプション）

### VPC Flow Logs を全 VPC で有効化する

デフォルトで無効。有効化しないと「どの IP がどこに通信したか」が永久に追えない。

```bash
aws ec2 create-flow-logs \
  --resource-type VPC \
  --resource-ids vpc-xxxxxx \
  --traffic-type ALL \
  --log-destination-type s3 \
  --log-destination arn:aws:s3:::your-flow-log-bucket
```

Security Lake を有効化していれば自動で収集される。

### GuardDuty のオプション保護を有効化する

デフォルトの GuardDuty は EC2・CloudTrail・DNS だけ。追加オプションで対象が広がる。

```bash
# S3 保護（S3 への不審なアクセスを検知）
aws guardduty update-detector \
  --detector-id <id> \
  --features '[{"Name":"S3_DATA_EVENTS","Status":"ENABLED"}]'

# EKS 保護・Lambda 保護・RDS 保護も同様に有効化
```

### アラート対応プレイブックを最低限書く

Slack にアラートが来たとき「で、どうするの？」が決まっていないと意味がない。最初は箇条書きでもよい。

- このアラートが来たら何を確認するか
- 誰が一次対応するか
- エスカレーション先はどこか

GuardDuty の finding タイプごとに AWS が推奨対応を公開しているので参考にする。

## Efficient：3〜6ヶ月でやること

「集めて通知する」から「調査できる・運用が回る」状態にする。

### Athena で Security Lake のログを分析できるようにする

Security Lake に溜まったログを SQL でクエリできる。「この IP が過去1ヶ月でどう動いたか」を調べられるようになる。

```sql
-- 例：特定 IP からの CloudTrail API 呼び出しを確認
SELECT eventtime, eventsource, eventname, sourceipaddress
FROM amazon_security_lake_glue_db_ap_northeast_1.amazon_security_lake_table_ap_northeast_1_cloud_trail_mgmt_1_0
WHERE sourceipaddress = '1.2.3.4'
  AND eventtime > now() - interval '7' day
ORDER BY eventtime DESC
```

インシデント調査の基本ツールになる。

### アラートのチューニング（false positive を削減する）

Security Hub の findings は最初から大量に出る。対応できないほど通知が多いとアラート疲れになり、本物の脅威を見落とす。

```bash
# 対応済み・意図的な設定は Suppressed にする
aws securityhub batch-update-findings \
  --finding-identifiers '[{"ProductArn":"...","Id":"..."}]' \
  --workflow '{"Status":"SUPPRESSED"}' \
  --note '{"Text":"意図的な設定のため抑制","UpdatedBy":"your-name"}'
```

月次でレビューして「常に出てるが問題ないもの」を仕分けていく作業。これを続けることで「鳴ったら本物」な状態に近づく。

### Inspector で ECR・EC2 を継続スキャンする

GuardDuty が「不審な挙動」を見るのに対して、Inspector は「既知の脆弱性（CVE）」を見る。

```bash
aws inspector2 enable \
  --resource-types EC2 ECR LAMBDA \
  --region ap-northeast-1
```

ECR にイメージを push するたびに自動スキャンが走り、CRITICAL な CVE があれば Security Hub に findings が上がる。

### 対応の一部を自動化する

アラートごとに人間が手で同じ操作をするのは続かない。繰り返し発生する対応は Lambda で自動化する。

```
例：
- 不審な IAM アクセスキーの検知 → 自動でキーを無効化
- 外部公開 S3 バケットの検知 → 自動でブロックパブリックアクセスをオン
- 感染が疑われる EC2 → 自動でセキュリティグループを隔離用に変更
```

全部自動化するのではなく「明らかに問題で、誤検知リスクが低い」ものから始める。

## Optimized：継続的にやること

この段階は専任のセキュリティエンジニアがいることが前提。いない場合は Efficient フェーズを磨き続けるほうが現実的。

### 検知ルールを Detection as Code で管理する

検知ルール（EventBridge ルール・OpenSearch クエリ・Lambda ロジック）を Git で管理して CI/CD でデプロイする。

- ルールの変更が追跡できる
- レビュープロセスを挟める
- 環境間でルールを一致させられる

### スレットハンティングを定期的に行う

アラートが来るのを待つのではなく、能動的に「怪しい動きがないか」を探す。月次や四半期で実施する。

Security Lake + Athena、または SIEM のクエリ画面で「過去30日間で普段と違う動きをした IAM ロールはあるか」「深夜に API を叩いているユーザーはいるか」のような仮説を立てて調べる。

### SIEM の導入を検討する（必要であれば）

Efficient フェーズを6ヶ月以上回して「GuardDuty の検知では足りない」「カスタムの相関ルールが必要」「スレットハンティングの効率を上げたい」という具体的な課題が見えてきたタイミングで検討する。

選択肢と判断軸は [ログ集約と SIEM 選定](ログ集約とSIEM選定.md) を参照。

## フェーズ別チェックリスト

### Quick Wins
- [ ] GuardDuty 有効化（全リージョン）
- [ ] Security Hub 有効化（FSBP 有効）
- [ ] CloudTrail 全リージョン証跡を確認
- [ ] Slack アラート通知設定（CRITICAL/HIGH のみ）

### Foundational
- [ ] Security Lake 有効化
- [ ] VPC Flow Logs 全 VPC で有効化
- [ ] GuardDuty オプション保護（S3・EKS・Lambda・RDS）
- [ ] アラート対応プレイブック（最低限）作成

### Efficient
- [ ] Athena で Security Lake クエリができる状態
- [ ] false positive の定期レビュー・抑制運用
- [ ] Inspector 有効化（ECR・EC2・Lambda）
- [ ] 高頻度対応の自動化（最低1〜2件）

### Optimized
- [ ] Detection as Code（GitOps 管理）
- [ ] 定期スレットハンティング（月次以上）
- [ ] SIEM 導入判断と運用
