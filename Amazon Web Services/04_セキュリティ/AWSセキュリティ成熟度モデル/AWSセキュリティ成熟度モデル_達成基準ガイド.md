# AWSセキュリティ成熟度モデル v2 達成基準ガイド

> 公式サイト: https://maturitymodel.security.aws.dev/en/model/
> アセスメントツール最新版: v7（[ダウンロード](https://maturitymodel.security.aws.dev/en/assessment-tools/)）

モデル自体は **v2**、セルフアセスメント用Excelツールが **v7**。
4つのフェーズ（Quick Wins → Foundational → Efficient → Optimized）× 10カテゴリで構成される。

以下、各項目について **0% / 25% / 50% / 75% / 100%** の5段階で達成基準を明文化する。
項目名はアセスメントツールv7日本語版の表記に準拠。

---

## 1. セキュリティガバナンス

### Phase 1: Quick Wins

#### セキュリティ連絡先の割当て

| 達成度 | 基準 |
|--------|------|
| 0% | 未設定、または退職者の個人メールが残っている |
| 25% | 一部アカウント（管理アカウント等）のみ設定済み。個人メールアドレスを使用 |
| 50% | 半数以上のアカウントに設定済みだが、DL（配布リスト）ではなく個人メール |
| 75% | 全アカウントに設定済みだが、DLでない or 2名未満のものがある |
| 100% | **全アカウント**にDL（2名以上）で設定。退職者なし。Security Hub **Account.1** が全アカウントでPASSED |

> **メンバーアカウントのみ（SCP/CT不可）：** 100%達成可能。アカウント設定 → 連絡先情報から設定できる。Organizations管理権限は不要。

#### リージョンの選択と無効化

| 達成度 | 基準 |
|--------|------|
| 0% | リージョン制限なし。どこでもリソース作成可能 |
| 25% | 承認リージョンを口頭/ドキュメントで決めただけ |
| 50% | SCP/Control Towerで一部リージョンをブロックしたが漏れがある |
| 75% | SCP/Control Towerで全未承認リージョンをブロック済みだが、既存の未承認リソースが残存 |
| 100% | **SCP or Control Towerで未承認リージョンをブロック**。未承認リージョンにリソースがゼロ |

> **プロジェクト内容を把握していないアカウント管理専任のセキュリティ担当の場合**
>
> 既存ワークロードに手が届かない立場でリージョン制限を達成するための現実的な選択肢。
>
> **選択肢1：SCP適用 + 事前影響調査セット（SCP権限がある場合の王道）**
> 1. AWS Config または Resource Explorer でリージョンごとのリソース分布を先に可視化
> 2. プロジェクト担当者へ「〇月〇日以降、承認外リージョンをSCPでDenyします」と事前通知
> 3. `aws:RequestedRegion` 条件で承認リージョン以外をDenyするSCPをOU単位で適用
> - 通知なしで即時適用するとプロジェクト側の既存デプロイが突然失敗するリスクがある
>
> **選択肢2：Control Towerガードレール適用（CT導入済みなら最速）**
> - CT管理コンソールから "Region deny" ガードレールを対象OUに有効化するだけで完了
> - SCPを自前で書く必要がなく、テスト済みのガードレールを使えるため安全
>
> **選択肢3：新規アカウントのみ先行適用（既存への影響リスクを避けたい場合）**
> - 新規作成アカウントにはSCPを即時適用し、既存アカウントは別途調査・通知後に段階適用
> - 既存は75%止まりでも、少なくとも新規アカウントで悪化を防げる
>
> **選択肢4：オプトインリージョンのAccount API無効化（補完的手段）**
> - `aws account disable-region` コマンドでオプトインリージョン（香港・バーレーン等）をアカウント単位で無効化できる
> - デフォルト有効リージョン（us-east-1、ap-northeast-1等）は無効化できないため、SCPの補完として使う
> - Organizations APIと組み合わせれば複数アカウントへの一括適用も可能
>
> **選択肢5：Configルール＋レポートのみ（SCP権限がない場合の現実解）**
> - SCPを触れる立場にない場合は、承認外リージョンにリソースが存在する場合にアラートするConfigルールを設定し、検知レポートをOrg管理者やプロジェクト側に連携する
> - 自分では止められないが「見えている」状態にすることで、75%相当の抑止力と報告体制を確立できる

> **メンバーアカウントのみ（SCP/CT不可）：** 上限25%相当。SCPがないとリージョンを強制的にブロックする手段がない。
> オプトインリージョン（香港・バーレーン等）は `aws account disable-region` で無効化できるが、デフォルト有効リージョン（ap-northeast-1等）はブロック不可。
> 「承認リージョンを文書で定義した」という状態（25%）が現実的な上限。Org管理者にSCP適用を依頼する稟議・要望書を作成することが実質的な打ち手。

### Phase 2: Foundational

#### コンプライアンスと規制要件の特定

| 達成度 | 基準 |
|--------|------|
| 0% | 適用される規制を把握していない |
| 25% | 主要な規制（PCI DSS、HIPAA等）の存在は認識しているが一覧化していない |
| 50% | 適用規制の一覧はあるが、具体的な管理策への落とし込みがない |
| 75% | 規制一覧＋管理策の対応表があるが、AWS共有責任モデル上の責任分界が曖昧 |
| 100% | **適用規制一覧が文書化**。各規制に対するAWS vs 自社の責任範囲が明確。**担当者がアサイン**され定期的にレビュー |

> **メンバーアカウントのみ（SCP/CT不可）：** 100%達成可能。ドキュメント作業・責任範囲の整理はAWS権限不要。

#### セキュリティ研修計画

| 達成度 | 基準 |
|--------|------|
| 0% | セキュリティ研修なし |
| 25% | セキュリティチームのみ個別に学習（計画なし） |
| 50% | 全従業員向けセキュリティ意識研修はあるが、ロール別の専門研修がない |
| 75% | ロール別研修計画（管理者/開発者/セキュリティ）が存在するが、受講率の追跡がない |
| 100% | **全従業員向け意識研修＋ロール別専門研修**の計画が存在。受講率を追跡。期限付き学習パス（Day1/Week1/Month1/3ヶ月）が設定されている |

> **メンバーアカウントのみ（SCP/CT不可）：** 100%達成可能。研修計画の立案・実施・追跡はAWS権限不要。

### Phase 3: Efficient

#### セキュアなアーキテクチャ設計

| 達成度 | 基準 |
|--------|------|
| 0% | アーキテクチャ図が存在しない |
| 25% | 概要レベルの構成図はあるがセキュリティ要素が反映されていない |
| 50% | セキュリティ要素（SG、サブネット分離等）を含む図があるがレビュー未実施 |
| 75% | セキュリティレビュー済みの図がある。ただし Fleet Manager移行やSGチェーン最適化が未完了 |
| 100% | **Workload Discovery on AWS等で可視化**。SGチェーンによるネットワーク経路最小化、Fleet Managerで管理ポート廃止が反映。エクスポート可能な文書として維持 |

> **メンバーアカウントのみ（SCP/CT不可）：** 100%達成可能。Workload Discovery on AWSや構成図ツールはアカウント内で利用可能。

#### Infrastructure as Code の活用

| 達成度 | 基準 |
|--------|------|
| 0% | 全て手動構築（マネコンポチポチ） |
| 25% | 一部のリソースをIaC化（カバレッジ25%以下） |
| 50% | 主要リソースはIaC化（カバレッジ50%程度）だがバージョン管理が不十分 |
| 75% | 大半のインフラがIaC＋バージョン管理済みだが、デプロイ前コンプライアンスチェックがない |
| 100% | **全インフラがCloudFormation/Terraform等でバージョン管理**。CloudFormation Guard等で**デプロイ前コンプライアンスチェックが自動実行** |

> **メンバーアカウントのみ（SCP/CT不可）：** 100%達成可能。CloudFormation/Terraform・GitHub Actions等のCI/CDはすべてアカウント内で構築できる。

#### タグ付け戦略

| 達成度 | 基準 |
|--------|------|
| 0% | タグ付けルールなし |
| 25% | 一部チームが独自ルールでタグ付けしている |
| 50% | 組織標準のタグ規約が文書化されているが、徹底されていない |
| 75% | タグ規約が徹底されており、コスト管理等に活用。ただしABACには未活用 |
| 100% | **標準化されたタグ規約が文書化**＋ABAC（属性ベースアクセス制御）に活用。タグ付け**自動化**が実装済み |

> **メンバーアカウントのみ（SCP/CT不可）：** 100%達成可能。タグ規約の文書化・Config/Lambdaによるタグ付け自動化・IAMポリシーを使ったABAC実装はすべてアカウント内で完結する。

### Phase 4: Optimized

#### セキュリティタスクの責任分担

| 達成度 | 基準 |
|--------|------|
| 0% | セキュリティはセキュリティチームだけの仕事 |
| 25% | 一部タスクは他チームと連携しているが場当たり的 |
| 50% | 主要なセキュリティタスクについてRACIを定義し始めている |
| 75% | RACIマトリクスが存在し運用されているが、サードパーティ契約への反映が未完了 |
| 100% | **全セキュリティタスクにRACIマトリクスが定義**。各部門にタスクが委譲。サードパーティ契約にも反映済み |

> **メンバーアカウントのみ（SCP/CT不可）：** 100%達成可能。RACI定義・サードパーティ契約への反映はAWS権限不要。

## 2. セキュリティ保証

### Phase 1: Quick Wins

#### クラウドセキュリティポスチャ管理 (CSPM)

| 達成度 | 基準 |
|--------|------|
| 0% | Security Hub未有効化 |
| 25% | 管理アカウントのみSecurity Hub有効化 |
| 50% | 複数アカウントで有効化済みだが組織全体ではない。標準チェックが一部のみ |
| 75% | 全アカウント有効化＋CIS/基礎ベストプラクティス有効。ただしスコアのKPI追跡がない |
| 100% | **全アカウント・全リージョンで有効化**。CIS/AWS基礎ベストプラクティスの標準チェック有効。スコアを**KPIとして追跡**し改善トレンドが出ている。専任の修復チームがアサイン |

> **メンバーアカウントのみ（SCP/CT不可）：** アカウント単体では100%達成可能。Security Hubの有効化・標準チェック・スコアのKPI追跡はすべてアカウント内で完結する。
> ただし評価基準の「全アカウント・全リージョン」という表現は満たせない。アセスメント時は「このアカウントに限ったスコープでの達成度」と明記する。

### Phase 2: Foundational

#### インベントリと設定モニタリング

| 達成度 | 基準 |
|--------|------|
| 0% | AWS Config未有効化 |
| 25% | 一部アカウントでConfig有効化のみ（ルール未設定） |
| 50% | 組織全体でConfig有効化。基本的なConfigルールを設定済み |
| 75% | Configルール＋SSM Automationで自動修復の一部を設定。ただし資産台帳が未整備 |
| 100% | **組織全体でConfig有効化**。Configルールでセキュリティベースライン検証＋SSM Automationで自動修復。**資産台帳**（重要度・ビジネスプロセス紐付け）が存在 |

> **メンバーアカウントのみ（SCP/CT不可）：** アカウント単体では100%達成可能。Config有効化・Configルール・SSM Automation自動修復・資産台帳整備はすべてアカウント内で完結する。
> 「組織全体で」という表現はスコープ外になる点を明記する。

### Phase 3: Efficient

#### コンプライアンスレポートの作成

| 達成度 | 基準 |
|--------|------|
| 0% | コンプライアンスレポートなし |
| 25% | Excelやドキュメントで手動作成。更新頻度が低い |
| 50% | Audit Managerを導入したが、カバーするフレームワークが限定的 |
| 75% | Audit Managerで主要フレームワークのレポートを生成できるが、担当者やプロセスが不明確 |
| 100% | **Audit Managerで自動化**。適用フレームワーク全てに対するレポートが生成可能。担当者が明確でプロセスが文書化 |

> **メンバーアカウントのみ（SCP/CT不可）：** 100%達成可能。Audit Managerはアカウント内で利用でき、フレームワークの選択・レポート生成・担当者設定はOrg管理不要。

### Phase 4: Optimized

#### 監査証拠の収集自動化

| 達成度 | 基準 |
|--------|------|
| 0% | エビデンスを全て手動収集 |
| 25% | 一部のエビデンス（CloudTrailログ等）は自動取得しているが大半は手動 |
| 50% | Audit Managerで主要エビデンスを自動収集。手動補完が多い |
| 75% | 大半のエビデンスが自動収集。手動アップロードは例外的なもののみ |
| 100% | **Audit Managerで自動収集が最大化**。CloudTrail・Security Hubから自動連携。**最小限の手作業**でレポート生成可能 |

> **メンバーアカウントのみ（SCP/CT不可）：** 100%達成可能。Audit Manager + CloudTrail + Security HubはすべてアカウントレベルのAPIで完結する。

## 3. ID とアクセス管理

### Phase 1: Quick Wins

#### 多要素認証 (MFA) の設定

| 達成度 | 基準 |
|--------|------|
| 0% | MFA未設定のrootアカウントが存在 |
| 25% | 全rootアカウントにMFA設定済みだが、仮想MFA（ソフトウェア） |
| 50% | rootにハードウェアMFA。特権IAMユーザーにもMFA設定済み |
| 75% | 全IAMユーザー/IdP連携ユーザーにMFA必須。ただしCognito等の顧客向けMFAが未対応 |
| 100% | **全rootにハードウェアMFA**。**全IAMユーザー/IdPユーザーにMFA必須**。顧客向けアプリにもMFA設定。Security Hub IAM.4 / IAM.9 / IAM.6 全PASSED |

> **メンバーアカウントのみ（SCP/CT不可）：** 100%達成可能。rootおよびIAMユーザーのMFA設定はアカウント内で完結。顧客向けCognito MFAもアカウント内で設定可能。

#### ルートの保護

| 達成度 | 基準 |
|--------|------|
| 0% | rootにアクセスキーが存在 or MFAなし |
| 25% | MFA設定済みだがソフトウェアMFA。アクセスキーが残存 |
| 50% | アクセスキー削除済み＋ハードウェアMFA。ただし監視やSCP未設定 |
| 75% | アクセスキー削除＋ハードウェアMFA＋CloudWatchアラーム設定済み。SCPでのメンバーアカウントroot制限が未完了 |
| 100% | **アクセスキー削除**＋**ハードウェアMFA**＋**rootログインのCloudWatchアラーム**＋**SCPでメンバーアカウントのroot操作を拒否**。IAM.4 / IAM.9 / IAM.6 / CloudWatch.1 全PASSED |

> **メンバーアカウントのみ（SCP/CT不可）：** 上限75%相当。アクセスキー削除・ハードウェアMFA設定・CloudWatchアラームはアカウント内で可能。
> ただし「SCPでメンバーアカウントのroot操作を拒否」はOrg管理者のみが設定できるため、この部分は達成不可。Org管理者への要請で対応する。

#### ID フェデレーション

| 達成度 | 基準 |
|--------|------|
| 0% | IAMユーザーで直接ログイン。長期アクセスキーを人間が使用 |
| 25% | IAM Identity Center（or 外部IdP）を導入済みだが、一部チームのみ利用 |
| 50% | 大半の従業員がSSO経由でログイン。ただし長期アクセスキーが一部残存 |
| 75% | 全従業員がSSO経由。長期アクセスキーなし。ただしHRシステムとの退職連動が未実装 |
| 100% | **全従業員がIAM Identity Center/外部IdPで認証**。**人間用の長期アクセスキーがゼロ**。**退職時に即アクセス失効**する仕組み（HR連動） |

> **メンバーアカウントのみ（SCP/CT不可）：** 既存IAM Identity Centerがorg側で設定済みなら「利用する」は可能。ゼロから構築する場合は管理アカウントが必要。
> 代替手段としてアカウントレベルのSAML連携（IAMコンソールのIDプロバイダー設定）でフェデレーション自体は実装できる。HR連動による即時アクセス失効は社内システム側の実装次第でOrg権限不要。

#### 意図しないアクセス権限の削除

| 達成度 | 基準 |
|--------|------|
| 0% | IAM Access Analyzer未設定 |
| 25% | 一部アカウントでAccess Analyzerを有効化したが放置 |
| 50% | 組織全体で外部アクセスAnalyzerが有効化。ただしFindingsのレビューが不定期 |
| 75% | Findingsを定期レビューし修復しているが、未使用のアクセスキー/権限の特定・削除が未完了 |
| 100% | **外部アクセスAnalyzerが組織全体に設定**。Findingsが**定期的にレビュー・修復**。未使用のアクセスキー/権限が**特定・削除済み** |

> **メンバーアカウントのみ（SCP/CT不可）：** アカウント単体では100%達成可能。アカウントレベルのAccess Analyzer（外部アクセス・未使用アクセス）はOrg管理不要で有効化・レビュー・修復ができる。
> 組織レベルアナライザー（全アカウントをカバー）はOrg管理が必要だが、自アカウント内はカバーされる。

### Phase 2: Foundational

#### 組織のアクセス許可ガードレール

| 達成度 | 基準 |
|--------|------|
| 0% | SCP/RCP未使用 |
| 25% | 基本的なSCPを1〜2個適用（例: リージョン制限のみ） |
| 50% | 主要なセキュリティ不変条件（root制限、リージョン制限等）をSCPでカバー |
| 75% | SCP＋RCPで大半のセキュリティ不変条件を強制。ただし文書化が不十分 |
| 100% | **セキュリティ不変条件がSCP/RCPで全て強制**。メンバーアカウント離脱防止、rootアクション制限、未承認操作防止等。**どのポリシーがどの脅威に対応するか文書化** |

> **メンバーアカウントのみ（SCP/CT不可）：** 上限25%程度。SCPとRCPはOrg管理者のみが設定できるため、メンバーアカウントから直接実装する手段がない。
> 現実的な打ち手：必要なSCPのポリシー文書とその根拠をOrg管理者に提出し、適用を依頼する。
> アカウント内の部分的な代替としてIAM Permission Boundaryで権限上限を設定する方法もあるが、Permission Boundary自体をIAM管理者が外せてしまうため強制力は弱い。

#### 一時的な認証情報の使用

| 達成度 | 基準 |
|--------|------|
| 0% | アプリケーションに長期アクセスキーをハードコード |
| 25% | 一部アプリケーションをIAMロール（Instance Profile等）に移行 |
| 50% | 主要アプリがIAMロールを使用。Secrets Managerを一部導入。ただしハードコードが残存 |
| 75% | 全アプリがIAMロール＋Secrets Manager。ただしコードリポジトリのシークレットスキャンが未実装 |
| 100% | **全EC2/Lambda/ECSがIAMロール**。**Secrets ManagerでDB認証情報を管理**。コードリポジトリに**シークレットスキャンが実装**。開発者トレーニング完了 |

> **メンバーアカウントのみ（SCP/CT不可）：** 100%達成可能。IAMロール・EC2 Instance Profile・ECS Task Role・Secrets Manager・コードリポジトリのシークレットスキャンはすべてアカウント内で完結する。

#### IMDSv2

| 達成度 | 基準 |
|--------|------|
| 0% | IMDSv1のインスタンスが多数存在 |
| 25% | 新規起動のデフォルトをv2に変更したが、既存は未対応 |
| 50% | 既存インスタンスの半数以上をv2に移行済み |
| 75% | ほぼ全インスタンスがv2。数台の残存あり |

## まとめ：フェーズ別の位置づけ

| フェーズ | 目安期間 | 概要 |
|----------|----------|------|
| **Quick Wins** | 1〜2週間 | 少ない工数で大きなセキュリティ効果。まずここから |
| **Foundational** | 1〜3ヶ月 | 基盤となる必須コントロール。ここまでが最低ライン |
| **Efficient** | 3〜6ヶ月 | スケーラブルな運用。セキュリティチームの効率化 |
| **Optimized** | 継続的 | 高度な自動化と専門チーム。成熟した組織の姿 |

### 達成度の読み方

| 達成度 | 意味 |
|--------|------|
| 0% | 未着手 or 著しく不十分 |
| 25% | 着手したが初期段階。一部のリソース/アカウントのみ対応 |
| 50% | 基本的な仕組みは入っているが、網羅性や運用プロセスに課題 |
| 75% | 大半が対応済み。残りは細かい最適化や文書化、モニタリングの強化 |
| 100% | 完全達成。技術的実装＋運用プロセス＋モニタリングが全て揃っている |

各項目の100%達成を目指すのではなく、**Phase 1 → 2 → 3 → 4の順に段階的に進める**のが推奨アプローチ。Quick Winsが全て100%になってからFoundationalに進む、というのが基本戦略。
