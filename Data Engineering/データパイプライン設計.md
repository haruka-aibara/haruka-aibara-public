# データパイプライン設計

「毎朝 ETL バッチが失敗していても誰も気づかない」「上流のデータ形式が変わったら下流が全部壊れた」——データパイプラインは作って終わりではなく、壊れにくく・失敗を観測できるように設計する必要がある。

---

## ELT vs ETL

| | ETL | ELT |
|---|---|---|
| 流れ | 変換してからロード | ロードしてから変換 |
| 変換場所 | 外部サーバー | データウェアハウス内 |
| 主な用途 | オンプレ DWH・機密データのマスキング | BigQuery・Snowflake・Redshift |
| 現代のトレンド | レガシー中心 | クラウド DWH では ELT が主流 |

クラウド DWH（BigQuery 等）は大量データの変換が安く速いため、**ロードしてから SQL で変換する ELT** が主流。

---

## レイヤー設計（Medallion Architecture）

```
Raw Layer（Bronze）: 外部データをそのまま保存
    ↓
Cleaned Layer（Silver）: クレンジング・型変換・重複排除
    ↓
Aggregated Layer（Gold）: BI・ML に向けた集計・結合済みテーブル
```

各レイヤーを分離することで:
- 上流データの問題調査が Raw に遡って確認できる
- Silver のロジック変更が Gold だけに影響を限定できる

## dbt でデータ変換を管理する

dbt（data build tool）は SQL の変換ロジックをコードとして管理し、依存関係の解決・テスト・ドキュメント生成を行うツール。

```sql
-- models/silver/orders_cleaned.sql
{{ config(materialized='table') }}

SELECT
  order_id,
  customer_id,
  CAST(order_date AS DATE) AS order_date,
  amount,
  -- 不正な金額を除外
  CASE WHEN amount > 0 THEN amount ELSE NULL END AS valid_amount
FROM {{ source('raw', 'orders') }}
WHERE order_id IS NOT NULL
```

```yaml
# models/silver/schema.yml
models:
  - name: orders_cleaned
    columns:
      - name: order_id
        tests:
          - not_null
          - unique
      - name: amount
        tests:
          - not_null
```

```bash
dbt run     # モデルを実行
dbt test    # テストを実行
dbt docs generate && dbt docs serve  # ドキュメントを生成・表示
```

## パイプラインのオーケストレーション

**Apache Airflow（DAG でワークフローを定義）:**

```python
from airflow import DAG
from airflow.operators.python import PythonOperator
from airflow.providers.google.cloud.operators.bigquery import BigQueryInsertJobOperator
from datetime import datetime, timedelta

with DAG(
    'daily_etl',
    schedule_interval='0 6 * * *',  # 毎朝 6 時
    start_date=datetime(2024, 1, 1),
    catchup=False,
    default_args={'retries': 2, 'retry_delay': timedelta(minutes=5)},
) as dag:

    extract = PythonOperator(
        task_id='extract_from_api',
        python_callable=extract_api_data,
    )

    load = BigQueryInsertJobOperator(
        task_id='load_to_bigquery',
        configuration={
            "load": {
                "destinationTable": {"projectId": "myproject", "datasetId": "raw", "tableId": "orders"},
                "sourceFormat": "NEWLINE_DELIMITED_JSON",
            }
        },
    )

    transform = PythonOperator(
        task_id='run_dbt',
        python_callable=lambda: os.system('dbt run --select silver.*'),
    )

    extract >> load >> transform
```

## データ品質チェック

```python
# Great Expectations でデータ品質を自動検証
import great_expectations as gx

context = gx.get_context()
ds = context.sources.add_pandas("my_datasource")
asset = ds.add_dataframe_asset("orders", dataframe=df)
batch = asset.get_batch_request()

suite = context.add_expectation_suite("orders_suite")
validator = context.get_validator(batch_request=batch, expectation_suite_name="orders_suite")

validator.expect_column_values_to_not_be_null("order_id")
validator.expect_column_values_to_be_unique("order_id")
validator.expect_column_values_to_be_between("amount", min_value=0, max_value=1_000_000)

results = validator.validate()
if not results.success:
    raise ValueError("Data quality check failed")
```

## 失敗したとき観測できるようにする

| 観測するべきこと | 方法 |
|---|---|
| パイプラインの実行ステータス | Airflow の UI・Slack 通知 |
| データ量の異常（0件・急増） | Great Expectations・dbt test |
| データの鮮度（更新が止まっている） | `_updated_at` カラムを監視 |
| 処理時間の増加 | Airflow のメトリクス → Grafana |

```sql
-- BigQuery でデータ鮮度を監視するクエリ
SELECT
  MAX(_updated_at) AS last_update,
  TIMESTAMP_DIFF(CURRENT_TIMESTAMP(), MAX(_updated_at), HOUR) AS hours_since_update
FROM `myproject.silver.orders`
```

`hours_since_update > 25`（期待値: 24時間）ならアラートを上げる。
