# 統合テスト

単体テスト（plan）では個々のリソースの設定しか確認できない。「VPC とサブネットと IGW が正しく繋がっているか」「セキュリティグループが意図どおりに通信を制御しているか」——こういった複数リソースの連携は実際に作らないとわからない。それが統合テストの役割。

---

## terraform test で統合テスト

`command = apply` にすることで実際にリソースを作成し、テスト後に自動で削除する。

```hcl
# tests/integration.tftest.hcl

variables {
  vpc_cidr = "10.0.0.0/16"
  env      = "test"
}

run "create_network" {
  command = apply

  assert {
    condition     = aws_vpc.main.id != ""
    error_message = "VPC が作成されていません"
  }

  assert {
    condition     = length(aws_subnet.public) == 2
    error_message = "パブリックサブネットが2つ作成されていません"
  }
}

run "verify_connectivity" {
  command = apply

  # 前の run で作成したリソースを参照できる
  assert {
    condition     = aws_internet_gateway.main.vpc_id == run.create_network.aws_vpc.main.id
    error_message = "IGW が VPC に紐づいていません"
  }
}
```

## Terratest（Go ベースのフレームワーク）

Gruntwork の Terratest は Terraform の統合テストによく使われる外部フレームワーク。

```go
package test

import (
  "testing"
  "github.com/gruntwork-io/terratest/modules/terraform"
  "github.com/stretchr/testify/assert"
)

func TestVpcModule(t *testing.T) {
  opts := &terraform.Options{
    TerraformDir: "../modules/networking",
    Vars: map[string]interface{}{
      "vpc_cidr": "10.0.0.0/16",
    },
  }

  defer terraform.Destroy(t, opts)
  terraform.InitAndApply(t, opts)

  vpcId := terraform.Output(t, opts, "vpc_id")
  assert.NotEmpty(t, vpcId)
}
```

## コスト管理

統合テストは実際のリソースを作るため課金が発生する。

- テスト専用の AWS アカウントを用意する
- `defer terraform.Destroy` で確実に削除する
- 高コストリソース（RDS, ElastiCache など）はモック・ローカル代替を検討
- CI では PR ごとではなく夜間実行やマージ後のみにする

`terraform test` は `command = apply` のブロックが終わると自動 destroy するので、テスト後の削除し忘れは防げる。
