# エンドツーエンドテスト

E2E テストはインフラを実際に構築し、アプリケーション視点での動作（HTTP レスポンス・DB 接続・API 疎通など）まで検証する。Terraform のテストの中で最もコストと時間がかかる。

---

## E2E テストの対象

- Web サーバーが 200 を返すか
- ロードバランサーのヘルスチェックが通るか
- データベースに接続できるか
- セキュリティグループが意図どおり通信を遮断しているか

---

## terraform test + external data での E2E

```hcl
# tests/e2e.tftest.hcl

run "deploy_all" {
  command = apply
}

run "verify_http" {
  command = apply

  # HTTP 疎通確認のために外部コマンドを使う
  module {
    source = "./tests/helpers/http_check"
  }

  variables {
    url = "http://${run.deploy_all.alb_dns_name}/health"
  }

  assert {
    condition     = module.http_check.status_code == 200
    error_message = "ヘルスチェックエンドポイントが 200 を返しません"
  }
}
```

## Terratest を使った E2E テスト

```go
func TestFullStack(t *testing.T) {
  opts := &terraform.Options{
    TerraformDir: "../",
  }

  defer terraform.Destroy(t, opts)
  terraform.InitAndApply(t, opts)

  albDns := terraform.Output(t, opts, "alb_dns_name")

  // HTTP 疎通確認（最大5分リトライ）
  url := fmt.Sprintf("http://%s/health", albDns)
  http_helper.HttpGetWithRetry(t, url, nil, 200, "OK", 30, 10*time.Second)
}
```

## テスト戦略のピラミッド

```
         /E2E\          ← 少数・高コスト
        /------\
       /  統合  \       ← 中程度
      /----------\
     /    単体    \     ← 多数・低コスト
    /  + validate  \
```

E2E テストは全部のリリースでは実行せず、ステージング環境への昇格時やリリース前のみ実行するなど、頻度を絞るのが現実的。

---

## CI での配置例

| ステージ | テスト種別 | タイミング |
|---|---|---|
| PR | validate + tflint + 単体（plan） | 毎 PR |
| main マージ | 統合テスト | main マージ後 |
| ステージングデプロイ後 | E2E テスト | ステージング確認時 |
