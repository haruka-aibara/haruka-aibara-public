# 状態ファイルの分割

すべてのリソースを1つの State に詰め込まない。適切に分割することでリスクと複雑さを下げる。

---

## 分割しないデメリット

- `terraform plan` が遅くなる（全リソースの状態を API で確認）
- 誰かが apply 中はロックがかかり、他の変更が全部ブロックされる
- 1つのミスが全インフラに影響する可能性がある
- 権限管理が難しい（全員が全リソースに触れる）

---

## 分割の単位

**レイヤー別**（よく使われる）

```
state/
├── foundation/    # VPC, IAM, DNS など変更頻度が低い基盤
├── database/      # RDS, ElastiCache など
└── application/   # ECS, ALB, Lambda など変更頻度が高い
```

**環境別**

```
state/
├── dev/
│   ├── terraform.tfstate
├── staging/
└── production/
```

**チーム別**

```
state/
├── platform/      # platform チームが管理
├── backend/       # backend チームが管理
└── frontend/      # frontend チームが管理
```

---

## S3 バックエンドでの分割設定

`key` を変えるだけでよい。

```hcl
# foundation/terraform.tf
terraform {
  backend "s3" {
    bucket = "my-tfstate"
    key    = "production/foundation/terraform.tfstate"
    region = "ap-northeast-1"
  }
}

# application/terraform.tf
terraform {
  backend "s3" {
    bucket = "my-tfstate"
    key    = "production/application/terraform.tfstate"
    region = "ap-northeast-1"
  }
}
```

---

## 分割したモジュール間のデータ参照

`data.terraform_remote_state` で別の State の出力値を参照できる。

```hcl
data "terraform_remote_state" "foundation" {
  backend = "s3"
  config = {
    bucket = "my-tfstate"
    key    = "production/foundation/terraform.tfstate"
    region = "ap-northeast-1"
  }
}

resource "aws_subnet" "app" {
  vpc_id = data.terraform_remote_state.foundation.outputs.vpc_id
}
```
