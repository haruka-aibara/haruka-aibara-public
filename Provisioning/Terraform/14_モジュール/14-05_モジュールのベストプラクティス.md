# モジュールのベストプラクティス

モジュールを使い始めると「大きすぎて再利用できない」「小さすぎて管理が煩雑」「provider をモジュール内に書いてしまって呼び出しが複雑」などの問題が出てくる。よく踏む罠とその対処をまとめる。

---

## 1. 適切な粒度で分割する

大きすぎるモジュールは再利用しにくく、小さすぎると管理が煩雑になる。

```
# 適切な例
modules/
├── networking/    # VPC, Subnet, Route Table, NAT Gateway
├── ecs-service/   # ECS Task Definition, Service, ALB Listener Rule
└── rds/          # RDS Instance, Parameter Group, Subnet Group
```

「このモジュールは何をするもの？」と一言で説明できる粒度が目安。

---

## 2. 変数には必ず description を書く

```hcl
variable "instance_type" {
  type        = string
  description = "ECS タスクの CPU/メモリ設定に合わせた EC2 インスタンスタイプ"
  default     = "t3.small"
}
```

モジュールのドキュメント代わりになる。

---

## 3. 出力は使う側が必要な値だけ公開する

内部実装の詳細まで全部 output に出さない。

```hcl
# ✅ 呼び出し元が必要なものだけ
output "cluster_id" { value = aws_ecs_cluster.main.id }
output "cluster_arn" { value = aws_ecs_cluster.main.arn }

# ❌ 内部の詳細まで全部出力しない
output "cluster_internal_setting" { value = aws_ecs_cluster.main.setting }
```

---

## 4. バージョンを明示する

公開モジュールを使う際は必ずバージョンを指定する。

```hcl
module "vpc" {
  source  = "terraform-aws-modules/vpc/aws"
  version = "~> 5.0"  # 必須
}
```

---

## 5. モジュール内で provider を設定しない

プロバイダーの設定は呼び出し元（ルートモジュール）で行い、モジュールには渡すだけにする。

```hcl
# ❌ モジュール内で provider を設定しない
provider "aws" {
  region = "ap-northeast-1"
}
```

---

## 6. README とサンプルを用意する

チームで使うモジュールには、最低限使い方の例と変数一覧を書いておく。`terraform-docs` を使うと variables.tf / outputs.tf から自動生成できる。

```bash
terraform-docs markdown . > README.md
```
