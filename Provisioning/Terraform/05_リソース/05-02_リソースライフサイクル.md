# リソースライフサイクル

Terraform のデフォルト動作は「変更があれば古いものを消して新しいものを作る」ことがある。本番データベースで何も考えずに apply したらインスタンスが消えた、というのは笑えない事故。`lifecycle` ブロックでこの挙動を制御できる。

---

## デフォルトの動作

```
plan → apply
  ├─ 新しいリソース         → Create
  ├─ in-place で変更できる  → Update
  ├─ in-place で変更できない → Delete → Create（replace）
  └─ .tf から消えた         → Delete
```

「in-place で変更できない」とは、クラウド側の制約でその属性は変更不可で再作成が必要なケース（EC2 の AMI 変更など）。

---

## lifecycle ブロックで挙動を変える

### create_before_destroy：先に新しいものを作る

デフォルトは「古いものを消してから新しいものを作る」。ALB のターゲットグループなど、依存リソースがある場合に削除が先に来ると困る。

```hcl
lifecycle {
  create_before_destroy = true  # 新しいものを作ってから古いものを消す
}
```

### prevent_destroy：誤削除を防ぐ

RDS や S3 など、消したら取り返しのつかないリソースに設定する。`terraform destroy` や `.tf` からの削除を試みるとエラーになる。

```hcl
lifecycle {
  prevent_destroy = true
}
```

### ignore_changes：特定の属性の変更を無視する

Terraform 管理外で変更される属性（ASG の desired_capacity など）を無視したいときに使う。

```hcl
resource "aws_instance" "web" {
  ami = "ami-0c55b159cbfafe1f0"

  lifecycle {
    ignore_changes = [ami]  # AMI を外部で変更しても Terraform は無視
  }
}
```

コード外での変更を許容する設定なので、使いすぎると State と実態がずれていく。本当に必要な場面に絞る。
