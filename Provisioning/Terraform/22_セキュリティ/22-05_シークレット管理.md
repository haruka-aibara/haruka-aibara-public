# シークレット管理

Terraform でシークレット（パスワード・API キー・証明書など）をどう扱うかは重要な設計判断。State ファイルに平文で記録されるリスクを理解した上で管理する。

---

## やってはいけないこと

```hcl
# ❌ 平文でハードコード
resource "aws_db_instance" "main" {
  password = "super-secret-password"   # git に上がる
}

# ❌ terraform.tfvars に書いてコミット
db_password = "super-secret-password"
```

---

## 環境変数から渡す

```bash
export TF_VAR_db_password="$(aws secretsmanager get-secret-value \
  --secret-id prod/db/password \
  --query SecretString --output text)"

terraform apply
```

`TF_VAR_<変数名>` の形式で環境変数を設定すると Terraform の変数として読み込まれる。

---

## AWS Secrets Manager からデータソースで取得

```hcl
data "aws_secretsmanager_secret_version" "db_password" {
  secret_id = "prod/db/password"
}

resource "aws_db_instance" "main" {
  password = data.aws_secretsmanager_secret_version.db_password.secret_string
}
```

Terraform 実行時に AWS API からシークレットを取得する。ただし **State ファイルに値が平文で記録される**ため注意。

---

## sensitive 変数

Terraform 0.14 以降、変数に `sensitive = true` を付けるとログ出力をマスクできる。

```hcl
variable "db_password" {
  type      = string
  sensitive = true
}
```

```
# apply 時のログ
+ password = (sensitive value)   # マスク表示
```

ただしこれはログのマスクであり、**State ファイルへの記録は防げない**。

---

## State ファイルの保護

シークレットは State に記録されるため、State 自体を保護する。

```hcl
terraform {
  backend "s3" {
    bucket  = "my-terraform-state"
    key     = "prod/terraform.tfstate"
    region  = "ap-northeast-1"
    encrypt = true   # S3 側で暗号化
  }
}
```

- S3 バケットのパブリックアクセスを無効化
- バケットポリシーで IAM 制限
- CloudTrail でアクセスログを記録

---

## パターン別の推奨アプローチ

| シークレットの種類 | 推奨管理方法 |
|---|---|
| DB パスワード | Secrets Manager からデータソースで取得 |
| API キー | CI の Secrets 変数 → `TF_VAR_*` 経由 |
| TLS 証明書 | ACM で管理。Terraform で参照のみ |
| SSH キー | Key Pair をコード外で管理 |
