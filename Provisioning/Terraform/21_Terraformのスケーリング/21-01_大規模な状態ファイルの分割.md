# 大規模な State ファイルの分割

1つの State ファイルに全リソースを詰め込むと、plan/apply が遅くなり、影響範囲も広くなる。State を適切に分割することが大規模運用の基本。

---

## 分割しないとどうなるか

- `terraform plan` が遅い（全リソースの差分を計算するため）
- 1つの変更が全リソースに影響するリスク
- チームの並列作業が困難（State ロックの競合）
- バグの影響範囲が大きい

---

## 分割の考え方

### レイヤーで分割（推奨）

```
terraform/
├── 01_network/       # VPC, サブネット, ルーティング
│   └── main.tf
├── 02_security/      # セキュリティグループ, IAM
│   └── main.tf
├── 03_data/          # RDS, ElastiCache
│   └── main.tf
└── 04_app/           # ECS, Lambda, ALB
    └── main.tf
```

ネットワーク層は変更頻度が低く、アプリ層は高い。頻度に応じて分けると apply の影響を局所化できる。

### 環境で分割

```
terraform/
├── dev/
├── staging/
└── production/
```

環境ごとに State を分離するのは最低限必要な分割。

### マイクロサービスで分割

サービスごとにリポジトリと State を分けるパターン。チーム自律性が高まるが State 間の依存管理が複雑になる。

---

## State 間のデータ共有

分割した State 間で値を共有するには `terraform_remote_state` を使う。

```hcl
# 04_app/main.tf

data "terraform_remote_state" "network" {
  backend = "s3"
  config = {
    bucket = "my-terraform-state"
    key    = "01_network/terraform.tfstate"
    region = "ap-northeast-1"
  }
}

resource "aws_ecs_service" "app" {
  # ネットワーク層の output を参照
  network_configuration {
    subnets = data.terraform_remote_state.network.outputs.private_subnet_ids
  }
}
```

---

## 分割の目安

| State あたりのリソース数 | 評価 |
|---|---|
| 〜50個 | 問題なし |
| 50〜200個 | plan が少し遅い。分割を検討 |
| 200個〜 | 分割を強く推奨 |

plan 時間が2分を超えてきたら分割のサインと考えてよい。
