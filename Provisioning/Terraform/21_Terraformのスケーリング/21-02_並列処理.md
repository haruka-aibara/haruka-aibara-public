# 並列処理

「apply が遅い」「AWS API の Throttling エラーが頻発する」——どちらも Terraform の並列処理の挙動と関係している。デフォルト動作を把握すると原因と対処がわかる。

---

## デフォルトの並列度

Terraform はデフォルトで最大 **10 並列**でリソースを処理する。

```bash
# 並列度を変更する
terraform apply -parallelism=20
```

---

## 依存グラフによる自動並列化

Terraform は内部で依存グラフ（DAG）を構築し、依存関係がないリソースを自動で並列実行する。

```hcl
# この3つは互いに依存がないので並列で作成される
resource "aws_s3_bucket" "logs" { ... }
resource "aws_s3_bucket" "assets" { ... }
resource "aws_s3_bucket" "backups" { ... }
```

```hcl
# aws_security_group が完成してから aws_instance が作られる（直列）
resource "aws_security_group" "web" { ... }

resource "aws_instance" "web" {
  vpc_security_group_ids = [aws_security_group.web.id]  # 依存あり
}
```

---

## 並列度を上げすぎると起きる問題

- AWS API のレート制限（Throttling）に引っかかる
- 同じリソースへの競合書き込みが発生する
- エラーの原因が追いにくくなる

デフォルトの 10 で問題ないケースがほとんど。API エラーが多発するなら下げる方向で調整する。

```bash
# API エラーが多い場合は並列度を下げる
terraform apply -parallelism=5
```

---

## 複数 State の並列実行

分割した State を並列で apply したい場合は、依存関係がないものをスクリプトや CI で並列実行する。

```bash
# ネットワーク層とセキュリティ層は依存がないので並列 OK
(cd 01_network && terraform apply -auto-approve) &
(cd 02_security && terraform apply -auto-approve) &
wait

# ネットワーク完成後にアプリ層を apply
(cd 04_app && terraform apply -auto-approve)
```

---

## `-target` で特定リソースだけ apply

巨大な State で特定リソースだけ先に作りたいときに使う。

```bash
terraform apply -target=aws_vpc.main -target=aws_subnet.public
```

`-target` は緊急時の手段。常用すると State の一貫性が崩れるリスクがあるため注意。
