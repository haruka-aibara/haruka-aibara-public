# HCP Vault をいつ使うか

## 問題

サービスが増えるにつれ、シークレットの置き場所がバラバラになっていく。

- DB パスワード → `.env` ファイル（Git に入れると終わり）
- CI/CD の認証情報 → GitHub Actions Secrets
- GCP の API キー → GCP Secret Manager
- Kubernetes のシークレット → Kubernetes Secrets
- 別チームへの共有 → Slack に貼る（最悪）

「どのシークレットがどこにあるか」を把握するのが難しくなり、誰がいつアクセスしたかも追えなくなる。

HCP Vault はシークレットを一箇所に集約し、アクセス制御・監査ログ・ローテーションをまとめて提供するマネージドサービス。

---

## HCP Vault vs 各種代替手段

| 手段 | 向いている場面 | 弱点 |
|------|--------------|------|
| GCP Secret Manager | GCP 内のサービス限定 | GCP 外から使いにくい、動的クレデンシャル非対応 |
| AWS Secrets Manager | AWS 内のサービス限定 | 同上 |
| GitHub Actions Secrets | CI/CD の認証情報 | アプリへの注入・ローテーションが手動 |
| Kubernetes Secrets | K8s Pod への注入 | 暗号化がデフォルト off、監査が弱い |
| **HCP Vault** | マルチクラウド・複数サービスを横断 | 学習コスト、コスト |

マルチクラウドや複数のサービス・チームをまたいでシークレットを一元管理したいときに Vault が刺さる。単一クラウドでシンプルな用途なら、そのクラウドのネイティブサービスで十分なケースも多い。

---

## HCP Vault（マネージド）vs セルフホスト Vault

自前で Vault クラスタを運用すると、HA 構成・ストレージバックエンド・アンシール・バックアップの管理が必要になる。HCP Vault はそれをすべて HashiCorp が管理する。

- **HCP Vault Dedicated**: フルマネージド。アップグレード・HA・バックアップは自動
- **HCP Vault Secrets**: さらに簡素化されたシークレット同期サービス（Vault の全機能は使えない）

プロダクションで動的クレデンシャルや細かいポリシー制御が必要なら Vault Dedicated。シークレットを安全に保管・同期するだけなら Vault Secrets でも十分。

---

## 動的クレデンシャルが Vault の真価

静的なシークレット（パスワードを安全に置く）だけなら GCP Secret Manager でも代替できる。Vault が特に強いのは**動的クレデンシャル**。

- Vault に「このロールで DB の一時ユーザーを発行して」と頼む
- Vault が DB に接続し、TTL 付きの一時ユーザー＆パスワードを発行する
- TTL が切れると自動で削除される

長期間有効なパスワードが存在しないのでリスクが根本的に下がる。AWS/GCP の IAM 認証情報も同様に動的発行できる。
