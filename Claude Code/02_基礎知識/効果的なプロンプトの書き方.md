# Claude Code で効果的なプロンプトを書くコツ

## 前提

Claude Code はチャットボットではなくエージェント。「質問に答えてもらう」のではなく「作業を依頼する」感覚で使うと、能力を引き出しやすい。

---

## 基本：具体的に頼む

### 悪い例

```
認証を直して
```

### 良い例

```
ログイン後にトークンが期限切れになったとき、自動でリフレッシュせずに
401エラーが返る問題を修正して。
src/lib/auth.ts の fetchWithAuth 関数が怪しい。
```

**ポイント：**
- 何が起きているか（症状）
- どうなるべきか（期待動作）
- どこが怪しいか（ヒント）

ヒントがなくても Claude は探してくれるが、あると無駄な探索が減ってトークンの節約になる。

---

## ファイルやコードを指して話す

Claude Code はプロジェクト全体を検索できるが、最初から対象を絞ると速い。

```
src/api/users.ts の getUser 関数に、キャッシュを追加して。
既存のキャッシュ実装は src/lib/cache.ts にある。同じパターンで。
```

「既存のパターンに合わせて」は非常に有効。Claude はコードベース内の既存パターンを読み取って従ってくれる。

---

## 段階的に進める

一度に大量の作業を頼むより、段階的に進めるほうが質が高い。

```
# ステップ 1
まず src/api/ 配下のエラーハンドリングの現状を調べて、問題点をリストアップして。

# ステップ 2（調査結果を見てから）
リストの 1, 3, 5 を修正して。2 と 4 はそのままでいい。
```

Plan モードを使って先に計画を立てさせるのも同じ考え方。

---

## やってほしくないことを書く

「やること」だけでなく「やらないこと」も伝えると事故が減る。

```
ユーザー一覧の API にページネーションを追加して。

やらないこと：
- 既存のレスポンス形式は変えない（後方互換を保つ）
- テストは既存のものを修正するだけで、新規追加はしない
```

---

## 画像を渡す

エラー画面やデザインカンプのスクリーンショットを貼れる。

```
この画面のレイアウト崩れを直して。
[スクリーンショット]
ヘッダーとサイドバーの間に隙間ができている。
```

「この画面」だけだと曖昧なので、何が問題なのかも言葉で補足する。

---

## コンテキストを活用する

### CLAUDE.md との組み合わせ

CLAUDE.md にプロジェクトのルールを書いておけば、プロンプトではタスクだけ書けばいい。

```
# CLAUDE.md に書いてあること
- Conventional Commits を使う
- テストは vitest で書く

# プロンプト（これだけでいい）
ユーザー削除の API を追加して、テストも書いて。
```

### 会話の流れを使う

Claude Code はセッション内の会話を覚えている。前の作業の文脈を活かせる。

```
（さっき作った API に）レート制限も追加して。
```

---

## アンチパターン

| やりがち | 改善 |
|---------|------|
| 「いい感じにして」 | 何をどうしたいか具体的に書く |
| 一度に 10 個の作業を頼む | 2〜3 個ずつ段階的に進める |
| エラー全文を貼らずに「エラーが出た」 | エラーメッセージをそのまま貼る |
| 修正箇所を指定せず「バグを直して」 | 再現手順や症状を伝える |

---

## まとめ

- 「何が」「どこで」「どうなるべきか」を具体的に書く
- 既存コードを指して「このパターンに合わせて」が有効
- 大きな作業は段階的に進める
- やらないことも明示する
- CLAUDE.md にルールを書いておけば、プロンプトは簡潔でいい
