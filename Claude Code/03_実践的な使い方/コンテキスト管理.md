# コンテキスト管理

## こういうときに困る

「セッションが長くなってきたら、前半で言ったことを Claude が忘れたり混乱したりするようになった」「トークンのコストが気になるが、どのタイミングでリセットすればいいかわからない」。

Claude Code はすべての会話履歴を毎回のリクエストに含めてモデルに送っている。セッションが長くなるほどトークン数が増え、コストが上がるだけでなく、パフォーマンスも低下する。これを「コンテキストの劣化（context rot）」と呼ぶ。

---

## コンテキスト管理の 3 つの手段

### 1. `/compact` — 要約して圧縮する

会話履歴を AI が要約し、重要な情報だけ残してコンテキストを圧縮するコマンド。

```
/compact
```

**向いている場面**：
- 作業の途中で「ここまでの状況を保ちながら続けたい」とき
- コンテキスト使用量の警告が出始めたとき

要約されるため一部の細部は失われるが、作業の継続に必要な情報は残る。

### 2. `/clear` — 完全リセットする

会話履歴をすべて削除してゼロからやり直す。

```
/clear
```

**向いている場面**：
- 別のタスクに移るとき
- 前のタスクの情報が新しいタスクに干渉しそうなとき
- セッションが大きく脱線してしまったとき

`/clear` 後は CLAUDE.md のみが読み込まれた初期状態に戻る。

### 3. セッションをそもそも短く設計する

コンテキスト管理の根本的な解決策は、1 セッションで抱えるタスクの量を減らすこと。

**実践的なやり方**：
- 1 セッション 1 タスクを原則にする
- 大きなタスクはサブエージェントに委譲してメインのコンテキストを汚染させない
- 完了したら `/clear` してから次のタスクに移る

---

## コンテキストの使用量を確認する

現在のコンテキスト使用状況はセッション画面に表示される。

一般的な目安：
- 80% を超えてきたら `/compact` を検討
- コンテキストが満杯になると Claude の回答品質が落ちる

---

## コンテキストが膨れやすいパターンと対策

| パターン | 原因 | 対策 |
|---------|------|------|
| 大量のファイルを読ませた | ファイル内容がすべてコンテキストに入る | 必要な部分だけ `@ファイル名:行番号-行番号` で指定する |
| エラーメッセージを何度も貼った | ログが長いとそのままコンテキストを消費 | エラーの要点だけを伝える |
| 長い会話を続けている | 往復ごとにトークンが積み重なる | 区切りごとに `/compact` or `/clear` |
| 不要な確認・雑談を挟んでいる | 作業と関係ない会話がコンテキストを圧迫 | 新しいタスクは新しいセッションで始める |

---

## セッションを再開する

`/clear` したあとや別日に作業を再開するとき、前の文脈を戻す方法は 2 つ。

**CLAUDE.md に書いておく**
繰り返し使う情報（プロジェクトの構成、コーディング規約、作業中のタスクの背景）は CLAUDE.md に書いておく。毎セッション自動で読み込まれる。

**`/resume`**
前回のセッション要約を読み込んで文脈を復元する（`/compact` で保存された要約がある場合）。
