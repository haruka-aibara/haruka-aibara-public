# CLAUDE.md の使い方

## CLAUDE.md とは

Claude Code が毎セッション開始時に自動で読み込む指示書。プロジェクトのルール、よく使うコマンド、コーディング規約などを書いておくと、毎回説明しなくても Claude が文脈を理解した状態で動いてくれる。

---

## どこに置くと何が起きるか

| 置き場所 | スコープ | 共有範囲 | 用途 |
|---------|---------|---------|------|
| `./CLAUDE.md` | プロジェクト全体 | チーム（git 管理） | プロジェクトの規約、ビルドコマンド、アーキテクチャ |
| `./.claude/CLAUDE.md` | プロジェクト全体 | チーム（git 管理） | 上と同じ（ディレクトリで整理したい場合） |
| `./.claude/rules/*.md` | プロジェクト全体（ファイルごとに条件付けも可） | チーム（git 管理） | トピックごとに分割したルール |
| `./CLAUDE.local.md` | プロジェクト（個人用） | 自分だけ（自動で .gitignore） | 自分の sandbox URL、テストデータなど |
| `~/.claude/CLAUDE.md` | 全プロジェクト共通 | 自分だけ | 個人の好み（コードスタイル、ツール設定など） |

**迷ったらプロジェクトルートに `CLAUDE.md` を 1 つ置くところから始めればいい。**

---

## 何を書くべきか

### 書くと効果が高いもの

```markdown
# プロジェクト概要
TypeScript + Next.js のフロントエンド。API は /src/app/api/ 配下。

# よく使うコマンド
- ビルド: `npm run build`
- テスト: `npm test`
- lint: `npm run lint`
- 単一テスト: `npm test -- --grep "テスト名"`

# コーディング規約
- 関数コンポーネントのみ使用（クラスコンポーネント禁止）
- スタイルは Tailwind CSS を使う（CSS Modules 禁止）
- API レスポンスの型は /src/types/ に定義する

# やってはいけないこと
- main ブランチに直接 push しない
- .env ファイルをコミットしない
- any 型を使わない
```

ポイントは **Claude が判断に迷いそうなことを先に書いておく** こと。

### 書いても意味が薄いもの

- 一般的なプログラミングの常識（「変数名はわかりやすく」等）
- Claude がコードを読めばわかること
- 頻繁に変わる情報（最新の Issue 番号など）

## rules/ でトピック分割する

プロジェクトが大きくなったら `.claude/rules/` でファイルを分けると管理しやすい。

```
.claude/
├── CLAUDE.md              # 全体共通のルール
└── rules/
    ├── frontend.md        # フロントエンド固有のルール
    ├── backend.md         # バックエンド固有のルール
    └── testing.md         # テストの書き方
```

### 特定のファイルにだけ適用する

YAML frontmatter で対象パスを指定できる。

```markdown
---
paths:
  - "src/api/**/*.ts"
---

# API ルール
- エンドポイントには必ずバリデーションを入れる
- エラーレスポンスは統一フォーマットを使う
```

この rules は `src/api/` 配下のファイルを触るときだけ読み込まれる。

---

## 他ファイルを読み込む（import）

`@パス` で別ファイルの内容を取り込める。

```markdown
プロジェクト概要は @README.md を参照。
利用可能なコマンドは @package.json を参照。
Git の運用ルールは @docs/git-workflow.md に従う。
```

README や既存ドキュメントの内容をコピペせずに参照できるので、二重管理にならない。

---

## 初期セットアップ

Claude Code のセッション内で `/init` を実行すると、プロジェクトを分析して CLAUDE.md を自動生成してくれる。

```bash
claude
> /init
```

生成された内容を確認・編集して使う。ゼロから書くより楽。

## まとめ

- まずはプロジェクトルートに `CLAUDE.md` を 1 つ置く
- 書くのは「よく使うコマンド」「コーディング規約」「やってはいけないこと」
- チームで共有するなら git 管理、個人用なら `CLAUDE.local.md`
- プロジェクトが大きくなったら `rules/` で分割する
