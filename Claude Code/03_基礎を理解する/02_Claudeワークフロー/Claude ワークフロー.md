# Claude ワークフロー

## なぜワークフローを意識するか

Claude Code に「このシステム全体をリファクタリングして」といきなり渡すと、途中で方向がぶれたり、関係ないファイルを変えたりすることがある。

Claude Code が内部的に動く仕組みを理解した上で、**Explore → Plan → Implement → Verify** の 4 フェーズを意識して使うと、大きなタスクでも一貫性のある結果が出やすくなる。

---

## 4 フェーズの流れ

### 1. Explore（探索）

まず Claude がコードベースを読んで状況を把握するフェーズ。

Claude は CLAUDE.md を読み込み、関連ファイルを辿って問題の全体像を把握する。このフェーズを丁寧にやるほど、後の計画・実装の精度が上がる。

**使えるアプローチ：**
- Plan Mode で読み取り専用のまま調査させる
- `@ファイル名` で調べてほしいファイルを明示する
- 「まず全体像を教えて」と最初に言う

### 2. Plan（計画）

探索で得た情報をもとに、変更の手順を設計するフェーズ。

何を・どの順番で・どのファイルに対して変えるかを明確にする。計画を出してもらってから承認することで、意図しない変更を防げる。

**使えるアプローチ：**
- Plan Mode で「この変更の計画を立てて」と依頼する
- 計画を確認してから実行させる（`~/.claude/plans/` に保存される）
- 不明な点を質問させてから進める

### 3. Implement（実装）

計画に沿ってファイルを変更していくフェーズ。

Claude は ファイル編集ツール・シェルコマンドを使って変更を実行する。権限設定に応じて確認プロンプトが出る。

**使えるアプローチ：**
- acceptEdits モードで小刻みに確認しながら進める
- サブエージェントに並列でサブタスクを委譲する
- 変更ごとに差分を確認する

### 4. Verify（検証）

変更後にテスト・lint・ビルドを実行して問題がないことを確認するフェーズ。

フックを設定しておくと、ファイル保存後に自動でテストや lint が走る。

**使えるアプローチ：**
- PostToolUse フックでファイル変更後に自動テストを実行する
- 「変更後にテストを実行して確認して」と最初から指示に含める
- CI/CD に Headless Mode で組み込む

---

## 実際の依頼の例

```
# 悪い例（いきなり大きな指示）
認証システム全体をリファクタリングして

# 良い例（フェーズを意識した依頼）
1. まず認証周りのコードをすべて読んで、現在の構造と問題点を教えて
   （Explore フェーズ）

→ Claude の分析を確認

2. その問題を解消するリファクタリングの計画を立てて。ファイルごとに
   何を変えるかリスト形式で出して
   （Plan フェーズ）

→ 計画を確認・承認

3. 計画通りに実装して
   （Implement フェーズ）

→ 変更を確認

4. テストを実行して確認して
   （Verify フェーズ）
```
