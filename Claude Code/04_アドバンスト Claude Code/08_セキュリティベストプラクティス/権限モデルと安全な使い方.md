# セキュリティベストプラクティス

Claude Code は強力なツールだが、ファイルを書き換えたり、コマンドを実行したりする権限を持つ。「うっかり本番データを消した」「マルウェア入りの外部コンテンツを実行させられた」という事故を防ぐために、最初から設定を整えておく。

---

## 権限の基本設計

Claude Code はデフォルトで読み取り専用。ファイル編集・コマンド実行など副作用のある操作は毎回確認を求めてくる。この確認を「毎回聞かれると邪魔」と感じて全部 allow に入れると、安全の仕組みが崩れる。

**やること：許可するものと絶対にブロックするものを明示する。**

`.claude/settings.json` に書く（プロジェクトルートに置くとチーム共有できる）：

```json
{
  "permissions": {
    "allow": [
      "Read",
      "Glob",
      "Grep",
      "Bash(npm test)",
      "Bash(npm run lint)",
      "Bash(git add *)",
      "Bash(git commit *)",
      "Bash(git status)",
      "Bash(git diff *)"
    ],
    "deny": [
      "Bash(git push --force *)",
      "Bash(rm -rf *)",
      "Bash(curl *)",
      "Bash(wget *)"
    ]
  }
}
```

- `allow` に入れたツール・コマンドは確認なしで実行される
- `deny` に入れたものは Claude が提案しても実行がブロックされる
- `Bash(パターン)` で特定コマンドだけ制御できる
- `deny` は `allow` より優先される

詳細な設定オプション → [Permissions](https://code.claude.com/docs/en/permissions)

## 書き込み範囲の制限

Claude Code が書き込めるのは、**起動したディレクトリとその配下だけ**。親ディレクトリへの書き込みは明示的な許可なしには行われない。読み取りはそれより広い範囲（システムライブラリ等）にアクセスできるが、変更は起動ディレクトリに閉じている。

**やること：プロジェクトのルートで起動する。** 意図しない範囲のファイルを触らせないために、`/` や `~` で起動しない。

---

## Sandbox でコマンド実行を隔離する

`/sandbox` コマンドを使うと、bash コマンドにファイルシステム・ネットワークの分離が適用される。Claude に自律的に動かせたいが範囲は限定したい、という場面に使う。

```
/sandbox
```

起動後は、Claude が実行するコマンドがサンドボックス内に閉じる。確認プロンプトの頻度も下がる。

詳細 → [Sandboxing](https://code.claude.com/docs/en/sandboxing)

---

## Devcontainer でさらに隔離する

本番環境に近い構成を触らせる場合や、信頼度の低いコードを扱う場合は devcontainer（開発コンテナ）を使う。コンテナ内で Claude Code を動かせば、ホストのファイルや認証情報への影響を切り離せる。

詳細 → [Development containers](https://code.claude.com/docs/en/devcontainer)

---

## プロンプトインジェクションへの対処

外部から取得したコンテンツ（Web ページ、Issue 本文、ログ等）に悪意ある指示が埋め込まれている場合、Claude がそれを「ユーザーの指示」と解釈してしまうリスクがある。これがプロンプトインジェクション。

**Claude Code が持っている保護：**

- `curl` / `wget` はデフォルトでブロックリストに入っている
- Web フェッチは独立したコンテキストウィンドウで処理される（メインのセッションに混入しない）
- 初回実行時・新しい MCP サーバー追加時は信頼確認が入る
- 怪しい bash コマンドは allowlist に入っていても手動承認を求める

**自分でやること：**

1. Claude が提案するコマンドを実行前に確認する
2. 信頼できないコンテンツを Claude に直接パイプしない（`cat 外部ファイル | claude` など）
3. 重要なファイルへの変更提案は特に慎重に確認する
4. 外部 Web サービスと連携するスクリプトは VM の中で動かす

## 認証情報を渡さない

Claude Code はセッション内の環境変数を読める。本番の API キーや DB パスワードを `.env` に入れたまま Claude に作業させると、誤ってコミットされたり、ログに出力されたりするリスクがある。

**やること：**

- `.env` に本番の認証情報がある場合は CLAUDE.md に「`.env` は読むな」と明記する
- CI/CD 環境では最小権限のトークンだけ渡す
- セッション中に `export` で本番クレデンシャルを設定しない

---

## MCP サーバーの扱い

MCP サーバーは外部ツールや API と Claude を接続する仕組みで、権限設定で制御できる。Anthropic は MCP サーバーを管理・審査していないため、**自分で書くか、信頼できるプロバイダーのものだけ使う**。

設定はソースコードの `.claude/settings.json` に入るので、チームのリポジトリで管理して共有する（どのサーバーを使っているか全員が確認できる状態にする）。

---

## チームでの運用

| やること | 方法 |
|---------|------|
| 組織標準の権限設定を強制する | managed settings（`~/.claude/settings.json` の組織配布）を使う |
| 誰が何をしたか把握する | OpenTelemetry メトリクスで Claude Code の利用を監視する |
| 設定変更を検知・ブロックする | `ConfigChange` フックを使う |
| 承認済み設定をチームで共有 | `.claude/settings.json` をバージョン管理に入れる |

詳細 → [Settings](https://code.claude.com/docs/en/settings)、[Monitoring usage](https://code.claude.com/docs/en/monitoring-usage)、[Hooks](https://code.claude.com/docs/en/hooks)

## 権限設定を確認する

現在どんな権限設定になっているかは `/permissions` コマンドで確認できる。

```
/permissions
```

定期的にチェックして、いつの間にか広すぎる権限が入っていないか見直す。

---

## 脆弱性を発見したら

Claude Code 自体にセキュリティの問題を見つけた場合は公開しない。[Anthropic の HackerOne プログラム](https://hackerone.com/anthropic-vdp/reports/new?type=team&report_type=vulnerability) から非公開で報告する。
