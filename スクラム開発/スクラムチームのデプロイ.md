# スクラムチームのデプロイ

「スプリントが完了した。でも本番には届いていない」——スプリントで「完成」させても、それが利用者の手に届かなければ価値を生んでいない。デプロイはスプリントの後に発生するオペレーションではなく、スプリントの定義完了（DoD）に含まれるべき活動だ。

---

## AI 時代のデプロイが抱える新しい問題

AI がコードを生成するようになって、デプロイに関する新しいリスクが増えた。

**Google DORA レポート 2024 の指摘：**

個人の生産性（AI で書くコードの量）は上がっているのに、組織のデリバリーパフォーマンス（リードタイム・デプロイ頻度・変更失敗率・MTTR）はほとんど変わっていない。

ボトルネックが「実装」から「レビュー・統合・テスト・デプロイ承認」に移っているからだ。AI でコードを速く書いても、その後の経路が詰まれば届く速さは変わらない。

---

## フィーチャーフラグ：「デプロイ」と「リリース」を分離する

AI が書いたコードが本番で壊れたとき、どう対応するか。

もっとも現実的な答えは**フィーチャーフラグ（機能フラグ）を使って「デプロイ」と「リリース（公開）」を分離すること**だ。

```
従来の考え方：
  main にマージ = 本番にデプロイ = ユーザーに届く

フィーチャーフラグを使う考え方：
  main にマージ → 本番にデプロイ（フラグ OFF 状態） → フラグを ON にしてリリース
```

フラグが OFF の状態でコードは本番サーバー上に存在しているが、ユーザーには見えない。問題が見つかればフラグを OFF に戻すだけ。コードのロールバックは不要。

**実績データ（LaunchDarkly 2024 年分析）：**
- フィーチャーフラグを採用したチームはリリースサイクルが **80% 短縮**
- デプロイ関連インシデントが **60% 減少**

---

## AWS でのフィーチャーフラグ実装

AWS AppConfig を使った最小構成：

```python
import boto3

appconfig = boto3.client("appconfigdata")

# セッションを開始
session = appconfig.start_configuration_session(
    ApplicationIdentifier="my-app",
    EnvironmentIdentifier="production",
    ConfigurationProfileIdentifier="feature-flags",
    RequiredMinimumPollIntervalInSeconds=30,
)

# 現在のフラグ設定を取得
config = appconfig.get_latest_configuration(
    ConfigurationToken=session["InitialConfigurationToken"]
)

import json
flags = json.loads(config["Configuration"].read())
new_feature_enabled = flags.get("new_checkout_flow", False)
```

AppConfig の設定で段階的なロールアウトもできる：

```json
{
  "new_checkout_flow": {
    "enabled": true,
    "percentage": 10
  }
}
```

最初は 10% のトラフィックで有効化し、問題なければ 50%、100% と増やす。CloudWatch アラームに連動した自動ロールバックも設定できる。

## スクラムチームの CI/CD パイプライン構成

典型的な構成：

```
GitHub Push（PR マージ後）
→ GitHub Actions / AWS CodePipeline（オーケストレーション）
→ AWS CodeBuild（ビルド・テスト・パッケージ）
→ Amazon ECR（コンテナイメージ登録）
→ Amazon ECS / Lambda（デプロイ）
→ CloudWatch Alarm（ヘルスチェック）
→ 問題があれば自動ロールバック
```

### AI コードに対して追加すべき CI チェック

AI が生成するコードには[固有のリスク](コードレビューはまだ必要か.md)がある。これをパイプラインで止める。

```yaml
# GitHub Actions の例
jobs:
  security-check:
    runs-on: ubuntu-latest
    steps:
      # 依存関係の脆弱性チェック（Slopsquatting 対策含む）
      - name: npm audit
        run: npm audit --audit-level=high

      # コンテナイメージの脆弱性スキャン
      - name: ECR image scan
        run: |
          aws ecr start-image-scan \
            --repository-name my-app \
            --image-id imageTag=${{ github.sha }}

      # SAST（静的アプリケーションセキュリティテスト）
      - name: Semgrep scan
        uses: returntocorp/semgrep-action@v1
```

Critical な CVE が検出された場合はデプロイを止める。AI が幻覚したパッケージ名が実在する悪意あるパッケージになっているリスクも、ここで検出できる。

## 定義完了（DoD）にデプロイを含める

スプリントの定義完了をコードベースで整合させる例：

```markdown
# Definition of Done チェックリスト

## コード品質
- [ ] PR がレビュー承認済み
- [ ] CI の全テストが pass
- [ ] npm audit / pip-audit でクリティカルな脆弱性なし

## デプロイ
- [ ] staging 環境にデプロイ済み
- [ ] 動作確認済み（手動 or 自動 E2E テスト）
- [ ] 本番デプロイ完了（フィーチャーフラグ OFF の状態でよい）

## モニタリング
- [ ] CloudWatch アラームが設定済み
- [ ] エラーレートのベースラインを確認
```

「フィーチャーフラグ OFF の状態で本番にデプロイ済み」を DoD に含めることで、スプリント終了後の「デプロイ作業」がなくなる。リリース（フラグを ON にする）はビジネス側の判断でいつでも実行できる。

---

## DORA メトリクスでチームの状態を測る

| メトリクス | エリートチームの水準 | 確認方法 |
|---|---|---|
| デプロイ頻度 | 1 日複数回 | CodePipeline 実行履歴 |
| リードタイム | 1 時間以内 | PR マージ〜本番デプロイの時間差 |
| 変更失敗率 | 5% 以下 | デプロイ後の rollback 率 |
| MTTR（平均回復時間） | 1 時間以内 | インシデント〜復旧の時間 |

AI ツール導入後にこれらの数字がどう変わったかを追うことで、「AI で速くなった」を感覚ではなくデータで評価できる。

2024 年の DORA レポートの発見を繰り返すと——個人の生産性は上がっていても、これらの指標は改善していないチームが多い。フィーチャーブランチが速くても、main に届くまでに詰まっている。
