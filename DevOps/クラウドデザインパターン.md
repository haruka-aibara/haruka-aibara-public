# クラウドデザインパターン

## 問題

クラウド上の分散システムは、ネットワーク遅延・部分的な障害・スケールの変動など、オンプレとは異なる失敗モードを持つ。毎回ゼロから対処法を考えるのは非効率で、車輪の再発明になりやすい。

主要なクラウドデザインパターンを知っておくと、「この問題、あのパターンで解ける」と引き出しが使えるようになる。

---

## 可用性・回復性パターン

### Circuit Breaker（サーキットブレーカー）

依存サービスへのリクエストが連続して失敗しているとき、しばらくの間リクエストを送るのをやめて（回路を切る）、無駄なタイムアウトの積み重ねを防ぐ。

```
通常時:  アプリ → 外部サービス（正常応答）
障害時:  アプリ → Circuit Breaker → [OPEN 状態] → 即エラーを返す
回復試行: 一定時間後に 1 リクエストを試して成功したら CLOSED に戻る
```

**状態遷移**: CLOSED（通常）→ OPEN（障害検知、即時フェイルバック）→ HALF-OPEN（回復確認）

AWS での実装例: App Mesh / Istio のトラフィックポリシー、Resilience4j（Java）、Lambda の retry 設定

### Retry（リトライ）

一時的な障害（ネットワーク瞬断など）は、少し待ってリトライすると成功することが多い。ただしリトライ間隔は **Exponential Backoff + Jitter**（ランダムな揺らぎ）を入れることで、全クライアントが同時にリトライしてサービスをDDOSしないようにする。

```python
# Exponential Backoff + Jitter の考え方
wait = min(base * 2^attempt, max_wait) + random(0, jitter)
```

### Health Endpoint Monitoring

アプリが「起動している」だけでなく「正常に動作している」かをチェックする専用エンドポイントを用意する。ロードバランサーやオーケストレーターがこのエンドポイントを定期的に叩いて死活監視する。

```
GET /health
→ { "status": "ok", "db": "ok", "cache": "degraded" }
→ HTTP 200（一部劣化でも稼働中）or 503（完全に落ちている）
```

## スケーラビリティパターン

### Queue-Based Load Leveling

突発的なトラフィックスパイクを吸収するために、リクエストをキューに溜めてバックエンドが処理できるペースで消化する。

```
リクエスト → SQS キュー → Lambda / ECS ワーカー（スケールアウト）
```

**向いている用途**: 画像処理・メール送信・非同期の重い処理。リアルタイム応答が不要な処理。

### Competing Consumers

複数のコンシューマーが同じキューを並列に処理する。コンシューマーを増減することでスループットをスケール調整できる。

```
SQS キュー ← ← 複数のワーカーが競合して取得
                Worker 1
                Worker 2
                Worker 3  ← Auto Scaling で増減
```

## 設計・実装パターン

### Sidecar

メインのアプリケーションコンテナの隣に補助的なコンテナを置いて、横断的な処理（ログ収集・プロキシ・設定取得など）を切り出す。

```yaml
# K8s Pod の例
spec:
  containers:
  - name: app          # メインアプリ
    image: myapp:1.0
  - name: log-shipper  # サイドカー（ログを Loki に転送）
    image: fluent-bit
  - name: envoy        # サイドカー（サービスメッシュ用プロキシ）
    image: envoy
```

アプリ側は横断処理を意識しなくていい。

### Strangler Fig（ストラングラーフィグ）

レガシーシステムをいきなり置き換えるのではなく、徐々に新システムに移行する。新機能から新システムで実装し、旧システムを少しずつ絞め殺す（strangler）。

```
Phase 1: 全トラフィック → レガシー
Phase 2: 新機能のみ → 新システム、既存機能 → レガシー
Phase 3: ほぼすべて → 新システム（レガシーは廃止）
```

## データ管理パターン

### CQRS（Command Query Responsibility Segregation）

書き込み（Command）と読み取り（Query）のモデルを分ける。読み取りが多い場合に読み取り専用のレプリカや専用モデルを最適化できる。

### Event Sourcing

状態を直接保存するのではなく「状態変化のイベント」を記録し、現在の状態はイベントを再生して導出する。監査ログが自然に手に入り、過去の任意の時点の状態を復元できる。

## パターン選択の考え方

| 問題 | 使うパターン |
|------|------------|
| 外部サービスが不安定でタイムアウトが積み重なる | Circuit Breaker |
| 突発的なスパイクでバックエンドが落ちる | Queue-Based Load Leveling |
| 横断的な処理（ログ、認証、プロキシ）をアプリに書きたくない | Sidecar |
| レガシーを止めずに新システムへ移行したい | Strangler Fig |
| ヘルスチェックを LB やオーケストレーターに使わせたい | Health Endpoint |

一度に全部覚えようとするより、今直面している問題に当てはまるパターンを調べる方が実践的。
