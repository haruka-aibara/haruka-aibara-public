# デプロイメント戦略——ダウンタイムなしでリリースする

「本番デプロイのたびに数分のメンテナンス時間が必要」「新バージョンで障害が起きたとき、全ユーザーに影響が出た」——デプロイの方法を変えるだけでこれらを解消できる。

---

## 戦略の比較

| 戦略 | ダウンタイム | ロールバック速度 | リスク | インフラコスト |
|---|---|---|---|---|
| In-Place（上書き） | あり | 遅い | 高 | 低 |
| Rolling Update | なし | 中程度 | 中 | 低 |
| Blue-Green | なし | 即時 | 低 | 2倍 |
| Canary | なし | 即時 | 最低 | 少し増える |

---

## Rolling Update（ローリングアップデート）

古いインスタンスを少しずつ新バージョンに入れ替えていく。Kubernetes のデフォルト。

```yaml
# Kubernetes Deployment
spec:
  replicas: 4
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1        # 同時に増やせる最大数
      maxUnavailable: 0  # 同時に落としてよい最大数（0=常に4台維持）
```

```
Before: [v1] [v1] [v1] [v1]
Step 1: [v1] [v1] [v1] [v2]
Step 2: [v1] [v1] [v2] [v2]
Step 3: [v1] [v2] [v2] [v2]
After:  [v2] [v2] [v2] [v2]
```

**メリット:** 追加インフラ不要。ほとんどの用途でこれで十分。
**デメリット:** v1/v2 が混在する時間帯がある。DB スキーマ変更は後方互換性が必要。

## Blue-Green デプロイ

Blue（現在）と Green（新バージョン）の2環境を用意し、ロードバランサーのトラフィックを切り替える。

```
                    ┌──────────────────────────────────┐
                    │         Load Balancer            │
                    └───────┬─────────────────┘
                    切り替え前: Blue (v1) 100%
                    切り替え後: Green (v2) 100%

Blue環境:  [v1] [v1] [v1]  （待機中）
Green環境: [v2] [v2] [v2]  （本番稼働中）
```

**AWS での実装例（ALB のターゲットグループ切り替え）:**

```bash
# Green 環境に新バージョンをデプロイしてヘルスチェック確認後
aws elbv2 modify-listener \
  --listener-arn arn:aws:elasticloadbalancing:... \
  --default-actions Type=forward,TargetGroupArn=arn:...:targetgroup/green/...
```

**ロールバック:** ターゲットグループを Blue に戻すだけ。数秒で完了。

**メリット:** 切り替えが瞬時、ロールバックも瞬時。
**デメリット:** インフラコストが2倍。DB スキーマ変更の扱いが複雑。

## Canary リリース

新バージョンへのトラフィックを少量（1〜5%）から徐々に増やす。問題があれば早期検知できる。

```
        ┌─────────────────────────────────────┐
        │            Load Balancer            │
        └──────┬───────────────────────┐
               │ 5%                    │ 95%
            [v2] (canary)           [v1] [v1] [v1]
```

**段階的なトラフィック移行:**

```
Phase 1: v2 に 5% → 15分監視（エラーレート・レイテンシを確認）
Phase 2: v2 に 25%  → 30分監視
Phase 3: v2 に 50%  → 30分監視
Phase 4: v2 に 100% → 完了
```

**AWS での実装（ALB の重み付きルーティング）:**

```json
{
  "Actions": [{
    "Type": "forward",
    "ForwardConfig": {
      "TargetGroups": [
        {"TargetGroupArn": "arn:...:v1", "Weight": 95},
        {"TargetGroupArn": "arn:...:v2", "Weight": 5}
      ]
    }
  }]
}
```

**Kubernetes での実装（Argo Rollouts）:**

```yaml
apiVersion: argoproj.io/v1alpha1
kind: Rollout
spec:
  strategy:
    canary:
      steps:
        - setWeight: 5
        - pause: {duration: 15m}
        - setWeight: 25
        - pause: {duration: 30m}
        - setWeight: 100
```

## DB スキーマ変更との組み合わせ

ダウンタイムなしデプロイの最大の難所。「Expand/Contract パターン」が定番。

```
Step 1 (Expand):   新カラム追加（NULLを許容）
                   → v1/v2 どちらでも動く状態にする

Step 2 (Migrate):  既存データをバックフィル
                   → 新カラムにデータを埋める

Step 3 (Contract): 旧カラムを削除
                   → v2 が完全に安定したあとに実施
```

「1デプロイで全部やろうとしない」のが原則。スキーマ変更とアプリ変更を別リリースに分ける。

---

## どの戦略を選ぶか

```
ダウンタイムが許容できる（深夜メンテ等）
  → In-Place で十分

24/7 のサービス、素早いロールバックが必要
  → Blue-Green

ユーザー影響を最小化しながら新機能を検証したい
  → Canary

Kubernetes を使っている・インフラを増やしたくない
  → Rolling Update（デフォルト）
```
