# 量子コンピュータの基礎 - なぜ今セキュリティエンジニアも知っておくべきか

「量子コンピュータは未来の話」と思っていると、暗号まわりで足元をすくわれる時代に入りつつある。本記事は専門家向けの解説ではなく、**インフラ・セキュリティ系の実務者が「最低限ここは押さえておきたい」レベル**でざっと整理するもの。

## なぜ今これを知っておくべきか

実用レベルの量子コンピュータはまだ存在しない。それでもセキュリティ側で話題になるのは、**「Harvest Now, Decrypt Later（HNDL）」**という攻撃シナリオが現実味を帯びてきたから。

- 攻撃者は今のうちに暗号化された通信・データを盗んで保存しておく
- 数年〜十数年後、量子コンピュータが実用化された時点で復号する
- 長期間秘匿が必要なデータ（医療・国家機密・知財・契約書）は **「今安全」では足りない**

つまり「いま量子コンピュータが完成していなくても、今送っているデータがいずれ読まれる」という前提で設計する話が出始めている。NIST が 2024年に耐量子暗号（PQC）の標準を正式公開したのも、この流れに乗ったもの。

## 古典コンピュータと何が違うのか

普通のコンピュータは「ビット」を使う。0 か 1 のどちらか。

量子コンピュータは「量子ビット（qubit）」を使う。これが古典ビットと違うのは次の2点。

**重ね合わせ（superposition）**: 0 と 1 の状態を同時に持てる。ただし「両方同時に計算できる」ではなく、「0 と 1 が確率的に共存する状態」を扱えるイメージ。観測した瞬間にどちらかに確定する。

**もつれ（entanglement）**: 複数の qubit を結びつけると、片方の状態を観測した瞬間にもう片方の状態も決まる。距離に関係なく相関する。

n 個の qubit があれば 2^n 個の状態を同時に扱える。これが「量子コンピュータが特定の問題で圧倒的に速い」と言われる理由。

## 何が得意で、何が得意じゃないか

ここを誤解すると「量子で全部が速くなる」と思ってしまうので注意。

**得意なもの（既存アルゴリズムが知られている）**:
- **整数の素因数分解**（Shor のアルゴリズム）→ RSA・楕円曲線暗号を破る
- **巨大なデータベースの探索**（Grover のアルゴリズム）→ 古典の半分くらいの計算量で済む
- **量子化学シミュレーション**（分子の振る舞いを直接計算）
- **特定の最適化問題**（厳密解ではなく近似解）

**得意じゃないもの**:
- 普通の Web アプリのレスポンス
- 機械学習の一般的な推論
- 動画エンコードや圧縮
- そもそも古典で十分速い処理

汎用 CPU の代わりではない。「特定の数学的構造を持つ問題」だけが劇的に速くなる、専用機に近い存在。

## セキュリティへの影響まとめ

実務に効く部分はここ。

| 暗号方式 | 量子の影響 | 対応 |
|---|---|---|
| RSA・DH・ECC（公開鍵） | Shor で破れる | PQC へ置き換え |
| AES（共通鍵） | Grover で実効鍵長が半減 | 鍵長を 2 倍にすればOK（AES-256 推奨） |
| ハッシュ（SHA-2/3） | 同上、強度が落ちる | 出力長を増やせばOK |

つまり**致命的なのは公開鍵暗号**。共通鍵・ハッシュは「鍵長を増やすだけ」で延命できる。

NIST が標準化した PQC アルゴリズム（2024年公開分）:
- **ML-KEM**（旧 CRYSTALS-Kyber） — 鍵交換
- **ML-DSA**（旧 CRYSTALS-Dilithium） — 署名
- **SLH-DSA**（旧 SPHINCS+） — 署名（ハッシュベース）

業界では「ハイブリッド方式」（既存の ECC と PQC を併用する）への移行が先行している。TLS 1.3 のハイブリッド鍵交換などはすでに実装が出ている。

## クラウド側の対応状況（2026年時点のざっくり感）

**AWS**:
- **KMS / ACM / Secrets Manager** などの API エンドポイントが PQC ハイブリッド TLS に対応済み
- **Amazon Braket** — 量子コンピュータ自体を使うサービス（IonQ・Rigetti・QuEra などの実機にアクセス）
- s2n（AWS の TLS 実装）が PQC ハイブリッドをサポート

**Azure**:
- **Azure Quantum** — 量子コンピューティング基盤
- 一部サービスで PQC 試験提供

**Google Cloud**:
- 社内 ALTS（内部通信プロトコル）の一部に PQC を導入済み

セキュリティ実務者として押さえるべきは「自分が使っているクラウドサービスが PQC 対応をいつ始めるか・していくか」のロードマップを定期的に確認すること。新しい暗号方式は**事業者側がインフラごと巻き取って差し替えてくれる**部分が大半なので、利用側はキャッチアップが中心になる。

## いま実務でやれること

「明日から量子コンピュータが来る」わけではないので、いきなり全部の暗号を差し替える必要はない。むしろやるべきはこの順番。

1. **長期秘匿が必要なデータの棚卸し** — HNDL が現実的に効くのはここだけ。10年以上保護が必要なものを特定する
2. **暗号アジリティの確保** — 暗号アルゴリズムを差し替えやすいコード・インフラ構成にしておく。ハードコードした RSA キーペアを直接使うような実装は今のうちに直す
3. **依存ライブラリ・通信プロトコルの把握** — どこで RSA/ECC を使っているか、棚卸ししておく（TLS・SSH・JWT 署名・コード署名・VPN など）
4. **クラウド事業者の PQC 対応をウォッチ** — KMS や ACM のリリースノートに「post-quantum」「PQC」が出てきたら確認

**「とりあえず PQC を入れる」はやらないこと**。標準化されたばかりで実装も枯れていないし、自前で PQC を扱うとパフォーマンス・互換性のトラブルを抱える。**事業者側の対応を待つ + 自前のコードは差し替え可能にしておく**が現実的。

## 量子コンピュータ自体を触ってみたい場合

- **IBM Quantum**（無料枠あり、Qiskit という Python SDK で書ける）
- **Amazon Braket**（AWS アカウントがあればすぐ試せる、課金あり）
- **Azure Quantum**

セキュリティ業務に直結はしないが、「量子回路ってこんな雰囲気か」を掴むだけでも、PQC 関連のニュースを読むときの解像度が上がる。
