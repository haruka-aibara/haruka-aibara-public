# S3 パブリックアクセスを組織全体で封鎖する

「S3 バケットを公開するつもりはなかったのに、設定を誤って全世界に公開していた」——これは AWS インシデントの中で最も件数が多いケースだ。

設定は 3 つの層に分かれている。**バケットレベル**だけ対策しても、**アカウントレベル**や**組織レベル**に穴があると防げない。逆に言えば、組織レベルから順に固めていけば、個別のバケット設定ミスが致命的な漏洩につながらない。

---

## 3 層の構造を理解する

```
組織レベル（SCP）
  ↓ Block Public Access の解除を禁止
アカウントレベル（S3 アカウント設定）
  ↓ アカウント内の全バケットにデフォルト適用
バケットレベル（個別バケット設定）
  ↓ バケット単位の設定
オブジェクト（公開可否の最終判定）
```

上位が Deny していれば、下位でどんな設定をしても公開にならない。

---

## 層 1：組織レベル — SCP で解除を禁止する

アカウントレベルの Block Public Access 設定を、メンバーアカウントが解除できないようにする。

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DenyS3PublicAccessModification",
      "Effect": "Deny",
      "Action": [
        "s3:PutBucketPublicAccessBlock",
        "s3:DeletePublicAccessBlock",
        "s3:PutAccountPublicAccessBlock"
      ],
      "Resource": "*",
      "Condition": {
        "StringNotEquals": {
          "aws:PrincipalOrgID": "${aws:PrincipalOrgID}"
        }
      }
    }
  ]
}
```

より強力に：設定変更自体を全面的に Deny する（管理アカウントから設定を集中管理する場合）。

```json
{
  "Sid": "DenyAllS3PublicAccessChanges",
  "Effect": "Deny",
  "Action": [
    "s3:PutBucketPublicAccessBlock",
    "s3:DeletePublicAccessBlock"
  ],
  "Resource": "*"
}
```

## 層 2：アカウントレベル — Block Public Access を全アカウントで有効化

### 単一アカウントに設定する

```bash
# アカウントレベルの Block Public Access を全項目 ON
aws s3control put-public-access-block \
  --account-id 123456789012 \
  --public-access-block-configuration \
    BlockPublicAcls=true,IgnorePublicAcls=true,BlockPublicPolicy=true,RestrictPublicBuckets=true
```

### 現在の設定を確認する

```bash
aws s3control get-public-access-block --account-id 123456789012
```

### 組織全アカウントに一括適用する（スクリプト）

```bash
#!/bin/bash
# 組織内の全メンバーアカウント ID を取得
ACCOUNT_IDS=$(aws organizations list-accounts \
  --query 'Accounts[?Status==`ACTIVE`].Id' \
  --output text)

for ACCOUNT_ID in $ACCOUNT_IDS; do
  echo "Setting Block Public Access for account: $ACCOUNT_ID"
  aws s3control put-public-access-block \
    --account-id "$ACCOUNT_ID" \
    --public-access-block-configuration \
      BlockPublicAcls=true,IgnorePublicAcls=true,BlockPublicPolicy=true,RestrictPublicBuckets=true 2>/dev/null || \
    echo "  FAILED (check assume role permissions)"
done
```

> このスクリプトを実行するには、各アカウントに対して `AssumeRole` できる権限が必要。  
> 実際には Organizations の `--target-id` を使った委任や、CloudFormation StackSets で一括適用するほうが現実的。

## 層 3：バケットレベル — 既存バケットの状況を把握する

アカウントレベルの設定を ON にすれば個別バケットの設定は上書きされるが、将来的にアカウントレベルが解除されたときのためにバケットレベルも設定しておく。

```bash
# アカウント内の全バケットの Block Public Access 設定を確認
for bucket in $(aws s3api list-buckets --query 'Buckets[*].Name' --output text); do
  status=$(aws s3api get-public-access-block --bucket "$bucket" \
    --query 'PublicAccessBlockConfiguration' 2>/dev/null || echo "NOT_SET")
  echo "$bucket: $status"
done
```

### パブリック公開になっているバケットを検知する

```bash
# AWS CLI でパブリックアクセス設定が緩いバケットを抽出
for bucket in $(aws s3api list-buckets --query 'Buckets[*].Name' --output text); do
  result=$(aws s3api get-bucket-acl --bucket "$bucket" 2>/dev/null | \
    grep -c "AllUsers\|AuthenticatedUsers")
  if [ "$result" -gt "0" ]; then
    echo "PUBLIC ACL DETECTED: $bucket"
  fi
done
```

Security Hub の **S3.1**（アカウントレベル Block Public Access）と **S3.2**（バケットにパブリックポリシーがある）が FAILED であれば即対応。

## CloudFormation StackSets で全アカウントに一括適用する

Organizations + StackSets を使えば、全アカウントへのアカウントレベル設定を自動化できる。

```yaml
# s3-block-public-access.yaml
AWSTemplateFormatVersion: '2010-09-09'
Resources:
  S3PublicAccessBlock:
    Type: AWS::S3::AccountPublicAccessBlock
    Properties:
      BlockPublicAcls: true
      BlockPublicPolicy: true
      IgnorePublicAcls: true
      RestrictPublicBuckets: true
```

```bash
# StackSet を作成して全アカウントに適用
aws cloudformation create-stack-set \
  --stack-set-name s3-block-public-access \
  --template-body file://s3-block-public-access.yaml \
  --permission-model SERVICE_MANAGED \
  --auto-deployment Enabled=true,RetainStacksOnAccountRemoval=false

aws cloudformation create-stack-instances \
  --stack-set-name s3-block-public-access \
  --deployment-targets OrganizationalUnitIds=r-xxxx \
  --regions ap-northeast-1 us-east-1
```

`SERVICE_MANAGED` モードにすると、Organizations に新規参加したアカウントへも自動適用される。

## 「これを適用すると業務が止まらないか？」の事前確認

Block Public Access を有効にすると、**静的ウェブサイトホスティングや公開が意図されたバケット**もアクセスできなくなる。

```bash
# 現在パブリックポリシーを持つバケットを事前に確認
for bucket in $(aws s3api list-buckets --query 'Buckets[*].Name' --output text); do
  policy=$(aws s3api get-bucket-policy --bucket "$bucket" 2>/dev/null)
  if echo "$policy" | grep -q "\*"; then
    echo "Has wildcard policy: $bucket"
    echo "$policy" | python3 -m json.tool | grep -A2 -B2 "\*"
  fi
done
```

CloudFront + S3 の構成なら、Block Public Access を有効にして OAC（Origin Access Control）経由でのみアクセスを許可するのが正解。パブリックアクセスは不要になる。

## まとめ

| 層 | 設定 | 効果 |
|---|---|---|
| 組織（SCP） | Block Public Access 変更を Deny | メンバーアカウントが解除できない |
| アカウント | Block Public Access 全項目 ON | アカウント内全バケットに適用 |
| バケット | 個別 Block Public Access ON | アカウント設定が解除されても保護 |

「組織レベルから順に固める」が鉄則。上位を固めれば、下位の設定ミスがあっても致命的な漏洩にならない。
